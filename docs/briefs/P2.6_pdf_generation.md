## Salut Codex

Sprint **livrable produit fini**. Tu transformes les .docx templates
+ les données métier (élève, entreprise, tuteur, dates, référentiel)
en **PDF prêts à signer**.

Sans ce sprint, PFMP Pilot affiche des données mais ne **livre pas**
de documents officiels — donc ne sert à rien dans la vraie vie d'un
DDFPT.

C'est un sprint **techniquement délicat** : génération PDF avec
champs dynamiques + tableaux (compétences) + signatures + cohérence
visuelle avec l'original Blaïse Cendrars.

---

## Objectif fonctionnel

Le prof référent (ou l'admin) peut :

1. Aller sur la fiche d'un élève
2. Cliquer "Générer la convention"
3. Le système :
   - Récupère le template du tenant (`establishment_formation_setup`)
   - Récupère les données : élève, classe, formation_referential,
     entreprise, tuteur, période, dates
   - Remplit les champs dynamiques détectés en P2.2
   - Génère un PDF
   - Le stocke dans `generated-pdfs` bucket
   - Affiche un aperçu + bouton "Télécharger" + "Envoyer au tuteur"
4. Idem pour le **livret PFMP** (avec les compétences à évaluer
   pré-listées depuis le référentiel)
5. Idem pour l'**attestation** (en fin de stage, après que le livret
   est rempli)

## Contraintes non-négociables

### 🛡️ Cohérence visuelle avec l'original

Le PDF généré doit **ressembler** au .docx original Blaïse Cendrars.
Pas une refonte design, pas un thème PFMP Pilot — c'est le **document
officiel du lycée**, on respecte sa charte.

Stratégie : convertir le .docx en HTML (via mammoth ou similaire),
remplacer les placeholders, puis convertir HTML → PDF (puppeteer ou
weasyprint).

### 🛡️ Tableaux compétences/AP

Le livret contient des tableaux d'attitudes/compétences avec **cases
à cocher** (Cocher une seule case par attitude évaluée).

Stratégie : générer ces tableaux **dynamiquement** en HTML/CSS à
partir du `formation_referential.attitudes_grid` et
`competences_grid` du tenant, puis les insérer dans le template.

### 🛡️ Champs dynamiques fiables

Si un champ détecté en P2.2 est `confidence: 'low'`, le PDF doit
quand même générer (pas de blocage), mais avec un placeholder visible
type `[À COMPLÉTER]` au lieu de la valeur. Ça permet à l'admin de
remplir manuellement.

### 🛡️ Pas de signature qualifiée

⚠️ Rappel VISION : ne jamais dire qu'un PDF généré est signé
électroniquement de manière qualifiée. Les signatures simples viennent
en P3.x via prestataire eIDAS.

Pour P2.6 : le PDF a juste des **emplacements de signature** vides
("Signature du proviseur", "Signature de l'entreprise", "Signature de
l'élève"). La signature simple via lien tuteur viendra en P3.

### 🛡️ Storage privé

Les PDF générés vont dans `generated-pdfs` bucket privé. Accès via
signed URL temporaire, pas accès public.

### 🛡️ Hash & traçabilité

Chaque PDF généré → :
- Calcul hash SHA-256 du contenu
- Sauvegarde dans table `documents` avec :
  - `student_id`, `period_id`, `document_type`
  - `storage_path`
  - `content_hash`
  - `generated_by`, `generated_at`
- Permet le **dossier de preuve** futur (P3.3)

## Architecture technique

### Choix de stack PDF

**Recommandation : conversion DOCX → HTML → PDF**

Pourquoi pas LibreOffice CLI (.docx → .pdf direct) :
- ❌ Lent (3-5s par PDF)
- ❌ Pas de contrôle fin sur le rendu
- ❌ Difficile de remplacer les champs dynamiques

Pourquoi DOCX → HTML → PDF :
- ✅ Mammoth.js extrait le HTML structuré
- ✅ JSDOM/Cheerio pour manipuler les placeholders
- ✅ Puppeteer (ou Playwright) pour convertir en PDF avec contrôle
  CSS
- ✅ Possibilité de générer dynamiquement les tableaux compétences

Pipeline :

```
.docx Blaïse Cendrars (template)
   │
   ▼ mammoth.convertToHtml()
   │
HTML structuré
   │
   ▼ Cheerio + remplacement placeholders + injection tableaux
   │
HTML personnalisé (élève + entreprise + référentiel)
   │
   ▼ Puppeteer.printToPDF()
   │
PDF final
   │
   ▼ Storage `generated-pdfs/{tenant}/{student}/{type}_{ts}.pdf`
   │
   ▼ INSERT documents (storage_path, content_hash, ...)
   │
PDF affiché à l'utilisateur via signed URL
```

### Edge Function `supabase/functions/generate-document/index.ts`

```typescript
serve(async (req) => {
  const { studentId, documentType, periodId } = await req.json()

  // 1. Auth
  // 2. Charger toutes les données nécessaires :
  //    - student + class + formation_referential
  //    - placement (entreprise + tuteur)
  //    - period
  //    - establishment (avec coordonnées)
  //    - establishment_formation_setup (pour template_path)

  // 3. Télécharger le template depuis storage
  const { data: templateBlob } = await supabase.storage
    .from('referentials-source')
    .download(setup.convention_template_path)

  // 4. mammoth → HTML
  const { value: html } = await mammoth.convertToHtml({
    arrayBuffer: await templateBlob.arrayBuffer()
  })

  // 5. Manipuler le HTML
  const $ = cheerio.load(html)
  const filledHtml = fillPlaceholders($, {
    eleve_nom: student.last_name,
    eleve_prenom: student.first_name,
    classe: class.name,
    entreprise_raison_sociale: company.name,
    entreprise_adresse: company.address,
    tuteur_nom: tutor?.last_name,
    tuteur_email: tutor?.email,
    tuteur_telephone: tutor?.phone,
    date_debut: period.start_date,
    date_fin: period.end_date,
    proviseur: establishment.proviseur_nom,
    assureur_nom: establishment.assureur_nom,
    assureur_numero: establishment.assureur_numero,
    // ... selon le mapping détecté en P2.2
  })

  // 6. Si livret : injecter les tableaux compétences/AP
  let finalHtml = filledHtml
  if (documentType === 'livret') {
    finalHtml = injectCompetencesGrid(filledHtml, formationReferential)
  }

  // 7. Puppeteer → PDF
  const browser = await puppeteer.launch()
  const page = await browser.newPage()
  await page.setContent(finalHtml)
  const pdfBuffer = await page.pdf({ format: 'A4', printBackground: true })
  await browser.close()

  // 8. Hash + storage
  const hash = await sha256(pdfBuffer)
  const storagePath = `${tenantId}/${studentId}/${documentType}_${Date.now()}.pdf`
  await supabase.storage.from('generated-pdfs').upload(storagePath, pdfBuffer)

  // 9. INSERT documents
  await supabase.from('documents').insert({
    establishment_id: tenantId,
    student_id: studentId,
    period_id: periodId,
    type: documentType,
    storage_path: storagePath,
    content_hash: hash,
    status: 'generated',
    generated_by: user.id,
  })

  // 10. Signed URL temporaire pour preview
  const { data: signed } = await supabase.storage
    .from('generated-pdfs')
    .createSignedUrl(storagePath, 3600)

  return Response.json({ documentId, signedUrl: signed.signedUrl, hash })
})
```

## ⚠️ Alternatives à considérer

Si Puppeteer en Edge Function s'avère trop lourd :

**Alternative 1 — Microservice PDF sur le VPS Hostinger** :
BraveHeart a un VPS (`187.124.50.143`). On peut y héberger un service
Node.js avec Puppeteer qui expose une API `POST /generate` appelée par
les Edge Functions. Avantages : pas de limite de taille, contrôle
total, cache local possible.

**Alternative 2 — Service externe Browserless.io** :
~$50/mois pour 10K PDF. Simple, robuste. Mais dépendance externe.

**Alternative 3 — Approche client-side** :
Génération PDF côté navigateur via jsPDF + html2canvas. Avantages :
zéro coût serveur. Inconvénients : qualité moindre, pas adaptée pour
livret 5+ pages.

**Mon conseil :** commencer par **Alt 1** (VPS BraveHeart) car
cohérent avec son écosystème. Si trop complexe → Alt 2.

À discuter dans ton plan avec BraveHeart.
