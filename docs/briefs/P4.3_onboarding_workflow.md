## Salut Codex

Sprint **scaling commercial**. Aujourd'hui, créer un nouveau tenant
demande à BraveHeart de bricoler en SQL Editor + Chrome MCP. Pas
scalable.

Avec ce sprint, BraveHeart **onboarde un nouveau lycée en 30 minutes**
via un workflow guidé : un lead converti → tenant créé → admin
identifié → analyseur IA lancé → lycée prêt à utiliser.

C'est **ce qui rend l'activité commerciale viable** au-delà du pilote
Blaïse Cendrars.

---

## Objectif fonctionnel

BraveHeart, depuis `/superadmin/onboarding`, peut :

1. Sélectionner un lead existant (table `lead_requests` de P4.2)
   ou créer manuellement
2. Workflow en 6 étapes (wizard) :

   **Étape 1 — Création tenant**
   - Pré-remplir depuis le lead
   - Slug auto-généré (ex : `lycee-charlemagne-paris`)
   - Subdomain auto : `<slug>.pfmp-pilot.fr`
   - Couleur primaire (héritée de défaut)

   **Étape 2 — Configuration coordonnées**
   - Tous les champs établissement (RNE, raison sociale, proviseur,
     assureur, téléphone, email, etc.)
   - Possibilité de copier-coller depuis le formulaire de demande
     (lead)

   **Étape 3 — Création de l'admin lycée**
   - Email + nom + prénom de l'admin (depuis le contact lead)
   - Génération mot de passe temporaire (envoyé par email)
   - Compte créé dans `auth.users` avec role `admin`

   **Étape 4 — Upload des documents officiels du lycée**
   - Convention vierge (.docx)
   - Livret PFMP (.docx) — un par formation enseignée
   - Attestation type (.docx)
   - Référentiel compétences si dispo

   **Étape 5 — Lancement Analyseur IA (Module 27)**
   - Pour chaque formation enseignée par le lycée :
     - Sélectionner le référentiel national parent (ou créer 100% local)
     - L'IA analyse les .docx, propose mapping
     - Validation/ajustement par BraveHeart inline
     - Sauvegarde référentiel local + setup
   - Cette étape réutilise totalement le code de P2.2 (Module 27)

   **Étape 6 — Récap + Activation**
   - Résumé de tout ce qui a été créé
   - Bouton "Activer le tenant" → `establishments.active = true`
   - Email de bienvenue à l'admin (lien login + identifiants)
   - Marquer le lead comme `converted`

3. À tout moment, peut sauvegarder en brouillon et reprendre plus tard
4. Une fois fini : le lycée client est **opérationnel sans
   intervention supplémentaire**

---

## Contraintes non-négociables

### 🛡️ Atomicité par étape

Chaque étape sauvegarde son état en DB (table `onboarding_sessions`).
Si BraveHeart ferme l'onglet, il peut reprendre où il en était.

### 🛡️ Vérification slug unique

Le slug du nouveau tenant doit être unique (pas déjà pris). Suggérer
des alternatives si conflit.

### 🛡️ Email admin unique

L'email de l'admin lycée ne doit pas exister déjà dans `auth.users`.
Sinon erreur claire.

### 🛡️ Permissions admin tenant

L'admin créé n'a accès **qu'à son tenant**. Pas d'accès cross-tenant
même temporaire.

### 🛡️ Audit log riche

Chaque action onboarding → audit log. Permet à BraveHeart de tracer
ce qui a été fait pour quel lycée.

### 🛡️ Rollback partiel possible

Si étape 5 plante (l'IA n'arrive pas à analyser un livret), BraveHeart
peut soit :
- Sauter l'étape (référentiel à compléter manuellement plus tard)
- Annuler l'onboarding (rollback : tenant supprimé, admin supprimé,
  retour lead `pending`)

## Schema additionnel

### Migration `supabase/migrations/0014_onboarding_sessions.sql`

```sql
create table if not exists onboarding_sessions (
  id uuid primary key default gen_random_uuid(),
  initiated_by uuid not null references profiles(id) on delete restrict,
  source_lead_id uuid references lead_requests(id) on delete set null,

  -- États du wizard
  step integer not null default 1 check (step between 1 and 6),
  status text not null default 'in_progress'
    check (status in ('in_progress', 'paused', 'completed', 'rolled_back')),

  -- Données accumulées par étape
  step1_data jsonb default '{}'::jsonb,
  step2_data jsonb default '{}'::jsonb,
  step3_data jsonb default '{}'::jsonb,
  step4_data jsonb default '{}'::jsonb,
  step5_data jsonb default '{}'::jsonb,
  step6_data jsonb default '{}'::jsonb,

  -- IDs créés (pour rollback)
  created_establishment_id uuid references establishments(id) on delete set null,
  created_admin_user_id uuid,
  created_referential_ids uuid[] default array[]::uuid[],

  started_at timestamptz not null default now(),
  completed_at timestamptz,
  updated_at timestamptz not null default now()
);

create index if not exists idx_onboarding_status on onboarding_sessions(status);
create index if not exists idx_onboarding_initiated_by on onboarding_sessions(initiated_by);

-- RLS : superadmin only
alter table onboarding_sessions enable row level security;
create policy "onboarding_sessions_superadmin" on onboarding_sessions
  for all using (current_role_is_superadmin()) with check (current_role_is_superadmin());
```

## Test obligatoire

Cycle complet end-to-end :

1. Lead de test créé via `/contact` (P4.2)
2. BraveHeart login superadmin
3. `/superadmin/onboarding` → "Onboarder ce lead"
4. Étape 1 : créer tenant `lycee-test-93`
5. Étape 2 : remplir coordonnées (faux RNE pour test)
6. Étape 3 : créer admin `admin@lycee-test.fr`
7. Étape 4 : uploader 1 livret factice (peut être un Word vide
   pour le test)
8. Étape 5 : lancer IA, valider mapping (peut être minimal pour test)
9. Étape 6 : activer
10. Vérifier en DB :
    - `establishments.active = true`
    - `auth.users` avec admin
    - `profiles` avec role `admin`
    - `formation_referential` local créé
    - `establishment_formation_setup` créé
    - `lead_requests.status = 'converted'`
11. Tester login admin avec mot de passe reçu par email
12. Vérifier qu'il accède bien à son tenant et **pas** aux autres
13. Cleanup (rollback session)
