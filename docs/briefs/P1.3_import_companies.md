## Migration schema nécessaire

Le schema actuel `companies` est trop minimaliste pour l'import
Blaïse Cendrars. Il faut ajouter :

```sql
alter table companies add column if not exists siret text;
alter table companies add column if not exists ape_code text;
alter table companies add column if not exists professional_family text;
alter table companies add column if not exists status text default 'active';
-- (active, partner, to_relaunch, to_watch, to_avoid)

create index if not exists idx_companies_city on companies(city);
create index if not exists idx_companies_family on companies(professional_family);
create index if not exists idx_companies_status on companies(status);
```

→ Crée la migration `0006_companies_enrichment.sql`.

## Familles de métiers — mapping

Le XLSX Blaïse a une colonne `Familles` avec des valeurs comme :
- "Commerce / relation client"
- "Automobile"
- "Maintenance industrielle"
- "Restauration"
- etc.

Mappe-les vers les enums VISION :
- commerce, gestion-administration, automobile, maintenance,
  electricite, numerique, batiment, logistique,
  hotellerie-restauration, sante-social, service-public, artisanat,
  autres

Si une famille du XLSX n'est pas dans l'enum → met-la dans `autres` et
log dans le rapport.

## Contraintes non-négociables

### 🛡️ Performance

646 entreprises = pas un volume énorme mais l'INSERT doit être en bulk
(1 query avec 646 rows ou batch de 100). Pas 646 queries.

### 🛡️ Dédoublonnage

Critère : `name + city` (lower-cased, stripped).

Comportement par défaut : skip doublons. Option "merger" disponible
mais cochée OFF par défaut.

### 🛡️ XLSX parsing

Utilise une lib légère : `sheetjs` (xlsx) ou `xlsx-populate`. Si le
projet a déjà une lib, réutilise.

### 🛡️ RLS

Admin/DDFPT only. `establishment_id` forcé tenant courant.

### 🛡️ Mode démo

Simulation mock comme P1.1.
