# P1.2.2 - Lien tenant + QR code etablissement

> Brief Codex - finition SaaS multi-tenant
> Date : 11 mai 2026
> Depend de : P1.2.1 en cours d'audit/merge

## Objectif fonctionnel

Chaque etablissement doit avoir un acces officiel partageable : un lien tenant clair, copiable, ouvrable et accompagne d'un QR code.

Le superadmin doit pouvoir remettre ce lien au lycee. L'admin/DDFPT doit aussi le retrouver dans ses parametres.

## Contexte

La base et le code connaissent deja l'identite multi-tenant :

- `establishments.slug`
- `establishments.subdomain`
- `establishments.custom_domain`
- `establishments.domain_verified`
- `VITE_PLATFORM_DOMAIN=pfmp-pilot.fr`
- `buildSubdomainUrl(slug)` dans `src/lib/tenant.ts`

Mais l'UX ne montre pas encore un acces etablissement utilisable. La fiche superadmin affiche seulement le slug/sous-domaine sous forme de champs techniques. Cote tenant, `/settings` ne donne pas encore au DDFPT le lien qu'il peut partager aux professeurs.

## Contraintes non-negociables

- Pas de migration SQL.
- Pas de modification RLS.
- Pas de generation de tenants par DNS automatique dans ce sprint.
- Pas de custom domain editable maintenant.
- Le mode demo doit rester intact.
- Le lien principal doit etre determine cote frontend a partir de l'etablissement :
  - si `custom_domain` est present et `domain_verified=true`, utiliser `https://custom_domain`;
  - sinon utiliser `https://<subdomain-ou-slug>.<VITE_PLATFORM_DOMAIN>`.
- Le QR code ne doit contenir que le lien tenant public.
- Le composant doit etre reutilisable sur la fiche superadmin et dans `/settings`.

## Perimetre precise

### 1. Helper tenant access

Creer un petit helper, par exemple `src/lib/tenantAccess.ts`, pour centraliser :

- `getTenantAccessSlug(establishment)`
- `buildTenantAccessUrl(establishment)`
- `buildTenantPlatformUrl(establishment)`
- `getTenantAccessStatus(establishment)`

Objectif : eviter de dupliquer la logique URL dans les routes.

### 2. Composant `TenantAccessCard`

Creer `src/components/TenantAccessCard.tsx`.

Le composant affiche :

- titre "Acces etablissement"
- URL principale en texte selectionnable
- badge de statut : sous-domaine plateforme ou domaine personnalise verifie
- QR code
- bouton "Copier le lien"
- bouton "Ouvrir l'espace"
- bouton "Telecharger le QR code" si techniquement simple

Le composant doit etre mobile-first : QR code centre, boutons empiles en mobile, ligne compacte en desktop.

### 3. Fiche superadmin

Modifier `src/routes/superadmin.establishments.$id.tsx` :

- afficher `TenantAccessCard` en haut de la fiche detail, juste apres les badges ou avant la carte "Identite tenant";
- conserver la carte "Domaines" technique pour audit, mais la rendre secondaire.

### 4. Settings tenant

Modifier `src/routes/settings.tsx` :

- afficher le meme `TenantAccessCard` pour `data.establishment`;
- l'admin/DDFPT voit ainsi son lien officiel a partager.

## Critères d'acceptation

- [ ] Superadmin voit un lien du type `https://slug.pfmp-pilot.fr` sur `/superadmin/establishments/$id`.
- [ ] Superadmin peut copier le lien.
- [ ] Superadmin peut ouvrir l'espace dans un nouvel onglet.
- [ ] QR code visible et lisible.
- [ ] Admin/DDFPT voit le meme lien dans `/settings`.
- [ ] Si un `custom_domain` verifie existe, il devient le lien principal.
- [ ] Si aucun custom domain verifie, le lien plateforme reste le lien principal.
- [ ] Mode demo ne casse pas.
- [ ] `npm run typecheck` OK.
- [ ] `npm run build` OK.

## Hors perimetre

- Ajout du wildcard DNS sur Vercel/OVH.
- Verification DNS automatique.
- Edition du custom domain par le DDFPT.
- Landing publique par sous-domaine.
- Redirection automatique apex -> tenant.
