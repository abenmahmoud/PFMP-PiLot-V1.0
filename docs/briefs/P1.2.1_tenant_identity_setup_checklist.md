## Salut Codex

P1.2 est mergé : un superadmin invite un DDFPT, l'email FR arrive, le DDFPT clique, définit son mot de passe via `/onboarding`, atterrit sur `/dashboard`.

Mais une fois là-bas, deux choses manquent pour qu'il se sente chez lui :

1. **Il ne peut pas modifier l'identité de son établissement** (nom, ville, UAI). `/settings` affiche ces champs en `ReadonlyField` sur la branche prod Supabase, alors que la démo permet l'édition. Quand un superadmin crée un tenant à la volée pour un pilote, le DDFPT doit pouvoir compléter / corriger ces infos lui-même.

2. **Il ne voit pas quoi faire ensuite.** Le dashboard affiche bien des empty states route-par-route ("Aucune période PFMP"), mais il manque une **vue d'ensemble du parcours de setup** ("3 étapes restantes sur 5 avant que ton tenant soit prêt").

Sprint volontairement petit : deux fichiers à toucher, deux services à étendre. Pas de nouvelle route, pas de nouvelle migration.

---

## Contexte

État vérifié au 11 mai 2026 :

- `src/routes/dashboard.tsx` : branche Supabase via `fetchDashboardData()`, gère le cas "pas de période" avec un EmptyState + CTA vers `/pfmp-periods`. Pas de vision agrégée du setup.
- `src/routes/settings.tsx` : branche Supabase. Édite `school_year`, `teacher_load_threshold`, `ai_enabled`, `rgpd_notice`, `logo_url`. **Mais affiche `name`, `city`, `uai` en `<ReadonlyField>`.** Le service `updateTenantSettings()` ne touche que `establishment_settings`, jamais `establishments`.
- `src/services/settings.ts` : `UpdateSettingsInput` ne contient pas `name | city | uai`.
- Base de données : `establishments(name, city, uai)` est UPDATE-able pour les rôles `admin | ddfpt` du tenant (RLS déjà vérifiée — `current_establishment_id()` JWT-aware).

---

## Objectif fonctionnel

Après ce sprint :

1. Un DDFPT (ou admin) peut éditer **nom, ville, UAI** de son établissement depuis `/settings`. Save → UPDATE `establishments` puis UPDATE `establishment_settings` (deux mutations en parallèle si besoin).
2. Sur `/dashboard`, **en haut de page**, une `SetupChecklist` affiche les 5 étapes de configuration tenant avec leur statut calculé en live :

| # | Étape | Condition de validation |
|---|---|---|
| 1 | Compte activé | toujours ✅ (on est connecté) |
| 2 | Identité établissement complétée | `establishments.city IS NOT NULL AND establishments.uai IS NOT NULL` |
| 3 | Première classe créée | `count(classes WHERE establishment_id = me) > 0` |
| 4 | Premier élève importé | `count(students WHERE establishment_id = me) > 0` |
| 5 | Première période PFMP définie | `count(pfmp_periods WHERE establishment_id = me) > 0` |

Étape non complétée → CTA cliquable (Link TanStack vers la route correspondante : `/settings`, `/classes`, `/students`, `/pfmp-periods`).

La checklist disparaît automatiquement quand les 5 étapes sont 🟢 (le DDFPT n'a plus besoin du guide).

---

## Contraintes non-négociables

### 🛡️ Mode démo intact
Le `SetupChecklist` doit fonctionner aussi en mode démo (5 étapes toutes vertes pour Jean Moulin). Pas de régression visuelle sur `dashboard.tsx` mode démo.

### 🛡️ Pas de migration SQL
Tout ce dont on a besoin existe déjà côté DB. Si tu identifies un besoin de migration, c'est un signal d'erreur d'analyse — viens en parler avant de coder.

### 🛡️ RLS naturelle
Tu n'ajoutes **aucune** policy. L'UPDATE sur `establishments` doit fonctionner pour `admin | ddfpt` via les policies existantes. Si ça échoue, c'est un bug RLS à remonter, pas à contourner.

### 🛡️ Atomicité du save Settings
Si UPDATE `establishments` réussit mais UPDATE `establishment_settings` échoue (ou inverse), affiche un message d'erreur clair et ne marque pas le formulaire comme "saved". Pas besoin de transaction Postgres (RPC trop lourd pour ce sprint) — fais les deux mutations en `Promise.allSettled`, traite les erreurs, affiche un toast.

### 🛡️ Performance checklist
Les 5 conditions doivent être calculées avec **maximum 4 count queries** (`students`, `classes`, `pfmp_periods`, et un SELECT identité déjà fait par `fetchDashboardData()`). Réutilise au maximum les données déjà fetchées par le dashboard plutôt que d'ajouter un appel séparé.

### 🛡️ Mobile-first
La checklist doit s'afficher proprement en 375px. Stack vertical, badges clairs, CTA accessibles au pouce.
