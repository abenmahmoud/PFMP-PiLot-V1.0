## Salut Codex

Sprint qui complète le **Module 26 — Espace paramétrage**. Tu construis
les **3 onglets restants** (Référentiels, Documents, Alertes) après
l'onglet Coordonnées (P2.4).

C'est ce qui rend le produit **vraiment durable** : un lycée configuré
une fois fonctionne en autonomie d'année en année, et peut affiner ses
référentiels au fil du temps.

## Objectif fonctionnel

L'admin (ou DDFPT) peut :

### Onglet "Référentiels" (`/settings/referentiels`)

1. Voir la liste des référentiels actifs pour son lycée
2. Pour chaque référentiel : statut (national hérité ou local
   personnalisé), nombre de classes liées, nombre d'AP, nombre de
   compétences
3. Cliquer sur un référentiel → ouvre une vue détaillée :
   - Liste des AP avec édition inline (label, niveaux)
   - Liste des compétences avec édition inline
   - Bouton "Créer une version locale" (si actuellement national hérité)
4. Si version locale créée : modifications appliquées uniquement au
   tenant, pas au national
5. Bouton "Réinitialiser au référentiel national" (si version locale)

### Onglet "Documents" (`/settings/documents`)

1. Voir la liste des templates uploadés par le superadmin (via P2.2)
2. Pour chaque type (convention, livret, attestation) : voir le
   template actuel, sa version, sa date d'upload
3. Bouton "Tester la génération" → ouvre un élève fictif pré-rempli,
   génère un PDF preview (mock pour l'instant, vraie génération en
   P2.6)
4. Bouton "Télécharger le template source" (re-download du .docx
   original)
5. **Note légale :** "Pour modifier les mentions légales d'un template,
   contactez le superadmin (champ sensible)"

### Onglet "Alertes" (`/settings/alertes`)

1. Voir la liste des seuils d'alerte configurables :
   - Seuil de surcharge professeur référent (par défaut 6 élèves)
   - Délai d'alerte visite tardive (par défaut 7 jours avant fin)
   - Délai d'alerte attestation manquante (par défaut 14 jours après
     fin)
   - Délai d'alerte convention non signée (par défaut 5 jours avant
     début)
2. Activer/désactiver chaque type d'alerte (toggle on/off)
3. Modifier les seuils

## Contraintes non-négociables

### 🛡️ Héritage propre des référentiels

Quand un admin clique "Créer une version locale" :
- Crée une nouvelle ligne `formation_referential` avec `source =
  'establishment'`, `parent_id = <id du national>`,
  `establishment_id = <tenant>`
- **Copie** les `attitudes_grid` et `competences_grid` du parent
- Met à jour les classes du tenant qui pointaient vers le national →
  pointent vers la version locale
- L'`establishment_formation_setup` du tenant est mis à jour aussi

À l'inverse, "Réinitialiser au référentiel national" :
- Supprime la ligne locale (CASCADE supprime aussi les pointeurs
  classes/setup)
- Crée des nouveaux liens classes → national parent

### 🛡️ Permissions

- **Lecture** : admin, DDFPT du tenant. Pas le PP, pas le référent.
- **Modification d'un référentiel local** : admin, DDFPT.
- **Modification d'un référentiel national** : superadmin only (via
  ses propres outils, pas accessible depuis cet écran).
- Templates : lecture seule pour l'admin lycée. Modification = champ
  sensible (P2.4 workflow).

### 🛡️ Traçabilité

Modifier un référentiel = audit log :
```
{
  action: 'referential_modified',
  target_type: 'formation_referential',
  target_id: <ref_id>,
  metadata: {
    field: 'attitudes_grid',
    diff: { ... },
    is_local: true
  }
}
```

### 🛡️ Pas de cassure de l'audit déjà fait

Si une visite a été validée avec un livret AP1-AP6, et que l'admin
**supprime** AP3 du référentiel… que se passe-t-il pour les visites
existantes ?

**Décision produit :** les visites validées sont **figées dans leur
contexte d'évaluation**. La table `visits` ou `visit_evaluations` doit
stocker une **copie** de la grille AP/compétences au moment de
l'évaluation.

→ **Pas dans ce sprint** (mais à mentionner dans le rapport comme
TODO P3 ou futur).

Pour P2.5 : afficher un warning à l'admin qui modifie un référentiel
dont des classes/visites dépendent.

## Schema additionnel

### Migration `supabase/migrations/0008_alert_thresholds.sql`

```sql
create table if not exists alert_thresholds (
  id uuid primary key default gen_random_uuid(),
  establishment_id uuid not null references establishments(id) on delete cascade,
  alert_type text not null,
  threshold_value integer,
  enabled boolean not null default true,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  unique (establishment_id, alert_type)
);

create index if not exists idx_alert_thresholds_estab on alert_thresholds(establishment_id);

-- RLS
alter table alert_thresholds enable row level security;

create policy "alert_thresholds_isolation"
  on alert_thresholds
  for all
  using (
    establishment_id = current_establishment_id()
    or current_role_is_superadmin()
  )
  with check (
    establishment_id = current_establishment_id()
    or current_role_is_superadmin()
  );

create trigger trg_prevent_alert_thresholds_estab_change
  before update on alert_thresholds
  for each row execute function prevent_tenant_change();

-- Constants pour seed des défauts
-- Pas seed initial : les défauts sont dans le code applicatif
-- jusqu'à ce que l'admin clique "Personnaliser"
```
