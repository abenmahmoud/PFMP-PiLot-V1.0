## Objectif fonctionnel

Après ce sprint :

1. La démo continue de fonctionner exactement comme avant
   (`VITE_DEMO_MODE=true`)
2. Une copie figée de `data/demo.ts` existe dans `data-reference/` —
   elle servira d'**objectif visuel** pour comparer prod vs ce qu'on
   doit reproduire
3. `useCurrentUser` est nettoyé : si Supabase est configuré et qu'un
   user est loggé en prod, il retourne le bon user ; sinon, il retourne
   le user démo
4. Toutes les pages qui s'attendent à des données affichent un **empty
   state** propre quand Supabase retourne 0 rows (pas de crash, pas de
   page blanche)

### Fichier 2 : `src/lib/useCurrentUser.ts` (à inspecter)

État actuel à confirmer (Sprint 2 a commencé à le bridger). Ce que tu
dois t'assurer :

- En mode démo (`VITE_DEMO_MODE=true`) : retourne le user démo
  sélectionné via le switcher rôle (comportement actuel)
- En mode prod (`VITE_DEMO_MODE=false` + Supabase configuré) :
  - Si pas de session Supabase → null/redirect vers `/login`
  - Si session Supabase → fetch `profiles` du user, retourne le shape
    cohérent avec ce qu'attendent les composants

Si le bridge est déjà OK → tu valides et tu documentes. S'il y a des
trous → tu les bouches.

## Pourquoi ce sprint en premier ?

C'est le préambule technique propre à la migration progressive démo →
Supabase. Sans ça :

- on n'a pas de référence visuelle figée pour comparer
- `useCurrentUser` reste un point fragile (Sprint 2 l'a partiellement
  câblé)
- les empty states manquent → P0.3+ vont crasher en prod vide

Avec ça : les 9 sprints suivants (P0.3 à P0.9) deviennent chirurgicaux.
