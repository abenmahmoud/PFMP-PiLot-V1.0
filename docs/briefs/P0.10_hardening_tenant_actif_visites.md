# P0.10 - Hardening tenant actif et visites

## Objectif fonctionnel

Avant P1 onboarding, verrouiller le contexte tenant actif : un superadmin ne doit jamais modifier "le premier etablissement par hasard", et les visites doivent utiliser le bon etablissement ou refuser proprement.

## Contexte

P0 final est merge sur `main` via PR #19. Les routes restantes lisent Supabase, `/visits/new` cree une visite brouillon, et le fallback demo superadmin a ete coupe hors mode demo.

Audit post-merge : `docs/sprints/P0_final_deep_audit.md`.

Probleme central :

- Les admins/DDFPT ont `profile.establishment_id`.
- Les superadmins ont `profile.establishment_id = null`.
- L'incarnation superadmin est portee par `auth.session.user.user_metadata.active_establishment_id`.
- Certains services P0 final lisent encore seulement `profile.establishment_id` ou fallback sur le premier etablissement.

## Scope IN

### 1. Ajouter un helper de contexte tenant actif

Fichier suggere : `src/lib/activeTenant.ts`

API attendue :

```ts
import type { AuthState } from '@/lib/auth'

export function getActiveEstablishmentId(auth: AuthState): string | null {
  if (!auth.profile) return null
  if (auth.profile.role !== 'superadmin') return auth.profile.establishment_id
  const metadata = auth.session?.user.user_metadata as Record<string, unknown> | undefined
  const active = metadata?.active_establishment_id
  return typeof active === 'string' && active.trim() ? active : null
}

export function isSuperadminImpersonating(auth: AuthState): boolean {
  return auth.profile?.role === 'superadmin' && Boolean(getActiveEstablishmentId(auth))
}
```

Notes :

- Le helper est un helper UX/contexte, pas une barriere de securite.
- La RLS et `current_establishment_id()` restent la verite.
