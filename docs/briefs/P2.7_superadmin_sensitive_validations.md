## Salut Codex

Sprint **final de P2**. Tu construis le **côté superadmin** du
workflow de validation des modifications sensibles initié en P2.4.

À la fin de ce sprint, le cycle complet fonctionne :
1. Admin lycée demande une modification (P2.4)
2. Superadmin voit la demande dans une file d'attente (P2.7)
3. Superadmin approuve ou rejette
4. La modification est appliquée + email envoyé à l'admin
5. Audit log complet

C'est aussi le **dernier sprint** avant que P2 soit done — et que le
pilote Blaïse Cendrars devienne **vraiment vendable** à d'autres
lycées.

---

## Objectif fonctionnel

Le superadmin (BraveHeart) peut :

1. Aller sur `/superadmin/sensitive-validations`
2. Voir 3 sections :
   - 🔵 **En attente** (badge avec compteur)
   - ✅ **Approuvées** (historique 30 derniers jours)
   - ❌ **Rejetées** (historique 30 derniers jours)
3. Pour chaque demande en attente :
   - Voir l'établissement demandeur (nom, slug)
   - Voir l'admin demandeur (nom, email, role)
   - Voir le **diff** : champ + ancienne valeur + nouvelle valeur
   - Voir le **motif** rédigé par l'admin
   - Voir la date de demande et l'âge ("il y a 2 heures")
4. Boutons d'action :
   - ✅ **Approuver** → applique la modification + email confirmation
   - ❌ **Rejeter** (avec motif obligatoire) → email rejet
   - 💬 **Demander des précisions** (envoie email à l'admin sans
     trancher)
5. Voir une **alerte** dans le dashboard superadmin si demandes
   anciennes en attente (> 24h)

## Contraintes non-négociables

### 🛡️ Superadmin only

Route + Edge Function vérifient le rôle. Pas d'accès admin lycée.

### 🛡️ Validation atomique

Approuver une demande = transaction atomique :
1. UPDATE `establishments` avec la nouvelle valeur
2. UPDATE `sensitive_change_requests` SET status = 'approved'
3. INSERT `audit_logs`
4. Trigger email
Si l'une échoue, **rollback complet**.

### 🛡️ Email envoyé à l'admin

À chaque approve/reject : email à l'admin demandeur.

**Pour ce sprint** : on peut **mocker l'email** en console.log + insert
dans une table `email_queue` (à traiter plus tard par un cron). Ne pas
brancher Resend/SendGrid maintenant — c'est P3 ou P4.

### 🛡️ Audit log riche

Chaque approve/reject :
```
{
  action: 'sensitive_change_approved',  // ou 'rejected'
  target_type: 'establishment',
  target_id: <establishment_id>,
  metadata: {
    request_id: ...,
    field: ...,
    old_value: ...,
    new_value: ...,
    reviewer_id: <superadmin>,
    review_motive: '...',  // si reject
  }
}
```

### 🛡️ Pas de modif rétroactive de documents

Si on change `proviseur_nom` aujourd'hui, les **PDF déjà générés** ne
sont **pas** modifiés (ce serait illégal de modifier un document signé
ou archivé). Seuls les **nouveaux PDF** auront le nouveau proviseur.

→ Ne pas s'occuper de ça dans P2.7 (c'est juste du UPDATE
`establishments`). Documenter dans le rapport comme TODO P3 si
pertinent (système de versions des templates).

### Fichier 4 : Edge Function `supabase/functions/process-sensitive-validation/index.ts`

Pourquoi une Edge Function plutôt qu'un appel direct côté client :
- L'opération est atomique (transaction multi-tables)
- Vérification rôle superadmin côté serveur (sécurité défensive)
- Préparation envoi email (futur)

```typescript
serve(async (req) => {
  // 1. Auth + check role superadmin
  // 2. Parse body : { requestId, action: 'approve'|'reject', motive? }
  // 3. Charger la demande + valider qu'elle est 'pending'
  // 4. BEGIN transaction
  // 5. Si approve :
  //    UPDATE establishments SET <field> = <new_value>
  // 6. UPDATE sensitive_change_requests SET status = ..., reviewed_by = user.id, ...
  // 7. INSERT audit_logs (riche)
  // 8. INSERT email_queue (à envoyer plus tard)
  // 9. COMMIT
  // 10. Retour { success: true, ... }
})
```

### Fichier 6 : Migration `supabase/migrations/0009_email_queue.sql`

Table simple pour stocker les emails à envoyer (cron plus tard) :

```sql
create table if not exists email_queue (
  id uuid primary key default gen_random_uuid(),
  to_email text not null,
  template_key text not null,
  template_data jsonb not null default '{}'::jsonb,
  status text not null default 'pending'
    check (status in ('pending', 'sent', 'failed')),
  sent_at timestamptz,
  error_message text,
  created_at timestamptz not null default now()
);

create index if not exists idx_email_queue_status on email_queue(status, created_at);

-- RLS : superadmin only
alter table email_queue enable row level security;
create policy "email_queue_superadmin_only" on email_queue
  for all using (current_role_is_superadmin()) with check (current_role_is_superadmin());
```

```sql

  from sensitive_change_requests
 where id = '<request_id>';

-- 2. L'établissement est mis à jour
select proviseur_nom from establishments
 where id = '<blaise>';
-- doit retourner "M. NOUVEAU"

-- 3. Audit log présent
select action, metadata from audit_logs
 where action = 'sensitive_change_approved'
 order by created_at desc limit 1;

-- 4. Email en queue
select * from email_queue
 where template_key like '%sensitive%'
 order by created_at desc limit 1;
```
9. Logout, login admin Blaïse
10. Vérifier que `/settings/coordonnees` affiche maintenant
    "M. NOUVEAU" et que la demande historique apparaît en
    "Approuvée"
11. Tester aussi le **rejet** : nouvelle demande, rejeter avec motif,
    vérifier que `establishments` n'a PAS changé
