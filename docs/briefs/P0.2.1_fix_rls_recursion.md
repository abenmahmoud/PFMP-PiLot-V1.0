# Brief — Sprint P0.2.1 : Fix RLS recursion sur `current_establishment_id()`

**Type :** Sprint fix bloquant (post-audit P0.2)
**Sprint parent :** P0.2 (validation prod, mergé via PR #8, verdict BLOCKER)
**Origine du bug :** Sprint 3 (`0004_rls_hardening.sql`) — fonction helper RLS définie en `SECURITY INVOKER` au lieu de `SECURITY DEFINER`
**Owner code :** Codex
**Auditeur :** Claude
**Reviewer final :** BraveHeart
**Branche cible :** `p0-2-1-fix-rls-recursion`
**Effort estimé :** XS (≤30 minutes incluant tests)
**Risque régression :** faible

---

## Contexte

Le rapport P0.2 (`docs/sprints/P0.2_rapport.md`) a confirmé end-to-end un bug bloquant :

- Toute lecture de `profiles`, `establishments`, `students` (et probablement d'autres tables filles) sous JWT `authenticated` retourne `HTTP 500` / Postgres `54001 stack depth limit exceeded`.
- L'UI prod login Auth réussit, mais l'application reste en **fallback démo permanent** (constat console : `[auth] fetchProfile: stack depth limit exceeded`).
- Le superadmin switcher est absent.
- Aucun parcours métier réel n'est utilisable, pour aucun rôle.

**Cause racine confirmée :** la fonction `public.current_establishment_id()` est définie en `SECURITY INVOKER` et lit `profiles`, alors qu'elle est invoquée depuis les policies `profiles_select`, `profiles_update`, `establishments_select`, `students_select`, `students_update`. La sous-requête `SELECT FROM profiles` re-déclenche `profiles_select`, qui re-évalue `current_establishment_id()`, etc. → récursion infinie.

Toutes les autres helpers (`current_role_app`, `is_superadmin`, `is_establishment_admin`, `is_referent_of`, `can_read_all_students`) sont correctement définies en `SECURITY DEFINER`. Le bug est strictement localisé à `current_establishment_id()`.

---

## Objectif

Corriger la fonction `public.current_establishment_id()` pour briser la récursion, sans modifier la sémantique métier.

---

## Scope IN

1. **Création d'une nouvelle migration** : `supabase/migrations/<YYYYMMDDHHMMSS>_fix_current_establishment_id_recursion.sql`
2. **Contenu de la migration** :

   ```sql
   CREATE OR REPLACE FUNCTION public.current_establishment_id()
   RETURNS uuid
   LANGUAGE sql
   STABLE
   SECURITY DEFINER
   SET search_path TO 'public'
   AS $function$
     select coalesce(
       case
         when current_role_app() = 'superadmin' then
           nullif(
             (auth.jwt() -> 'user_metadata' ->> 'active_establishment_id'),
             ''
           )::uuid
         else null
       end,
       (select establishment_id from profiles where id = auth.uid())
     );
   $function$;
   ```

3. **Application** de la migration via `supabase db push` (en local d'abord, puis en prod via Supabase MCP `apply_migration` si Codex préfère, sinon Adel via `supabase db push`).
4. **Validation post-fix** (définition de done — voir section dédiée).

## Scope OUT (NE PAS toucher)

- ❌ Ne PAS modifier les autres fonctions helper (`current_role_app`, `is_superadmin`, `is_establishment_admin`, `is_referent_of`, `can_read_all_students`) — elles sont déjà correctes.
- ❌ Ne PAS modifier les policies existantes — la simple correction de `current_establishment_id()` casse mécaniquement la boucle pour toutes les policies qui l'invoquent.
- ❌ Ne PAS modifier `0004_rls_hardening.sql` (migration historique mergée) — créer une nouvelle migration au-dessus.
- ❌ Ne PAS toucher au code applicatif (TypeScript, React, Next.js) — fix purement DB.
- ❌ Ne PAS recréer `user_roles`, `rls_audit`, le trigger `prevent_establishment_id_change` (déjà existants).
- ❌ Ne PAS introduire d'élévation de privilège : la sémantique du fallback (`establishment_id` du user authentifié) reste identique.
