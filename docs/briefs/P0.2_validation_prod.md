## Périmètre précis

### Étape 1 — Test login en prod

1. Va sur l'URL prod Vercel (vérifie en console : la dernière URL est
   probablement `https://pfmp-pi-lot-v1-0.vercel.app` ou un alias)
2. Tente login avec `test@pfmp-pilot.fr` / `TestPFMP2026!`
3. Documente :
   - Login réussi ? oui/non
   - Redirection après login : où ?
   - Présence du dropdown superadmin tenant switcher : oui/non
   - Empty states ou crashes sur les pages ? screenshot si crash

### Étape 2 — Test logout

1. Clique sur "Se déconnecter" / topbar logout
2. Documente :
   - Redirection vers /login : oui/non
   - Pas de boucle de rechargement : confirme
   - localStorage cleané : oui/non (vérif DevTools)

### Étape 3 — Tests RLS isolation

Le script `supabase/tests/rls_isolation.sql` existe (Sprint 3) avec 5
tests. Il faut :

1. Créer 2 establishments fictifs (ex : "Test Tenant A", "Test Tenant
   B") via SQL Editor
2. Créer 2 users (1 par tenant) via Supabase Auth UI
3. Promouvoir chaque user admin de son tenant
4. Lancer le script `rls_isolation.sql` (adapte les UUIDs si nécessaire)
5. Vérifier que les 5 tests passent verts
6. **Cleanup** : supprime les 2 tenants de test + 2 users à la fin

Si tu détectes une fuite (test rouge), STOP, documente précisément, on
corrige avant de continuer.

### Étape 4 — Validation des 26 tables protégées

Lance dans SQL Editor :

```sql
select count(*) as unprotected_tables
  from rls_audit
 where rls_enabled = false or policy_count = 0;
```

Doit retourner **0**. Sinon tu identifies quelle table fuit.
