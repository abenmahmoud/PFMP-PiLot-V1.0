## Salut Codex

Première **mutation Supabase réelle** dans une route métier. Un prof
fait une visite, remplit le formulaire mobile, ça s'enregistre en DB.

C'est le moment de validation fonctionnelle de toute la chaîne :
auth → RLS → mutation → audit log.

---

## Objectif fonctionnel

Un prof référent peut :

1. Cliquer "Nouvelle visite" depuis la fiche d'un de ses élèves
2. Arriver sur `/visits/new?studentId=XXX` avec form pré-rempli
   (élève, entreprise, tuteur déduits)
3. Remplir le formulaire mobile-first :
   - Date (par défaut aujourd'hui)
   - Type contact (visite sur site / appel / visio / email)
   - Élève présent (oui/non)
   - Tuteur rencontré (oui/non)
   - Conditions de stage (texte court)
   - Activités réalisées (texte)
   - Posture professionnelle (échelle ou texte)
   - Points positifs
   - Difficultés
   - Remarque tuteur
   - Remarque professeur
   - Niveau alerte (info/vigilance/problème/urgent)
   - Prochaine action
4. Sauvegarder en brouillon (à tout moment)
5. Soumettre quand prêt → statut "brouillon" en DB
6. Audit log automatique

P0.7 ne fait **PAS** la validation du CR (passage brouillon → validé).
Ça vient en P0.8.

---

## Contraintes non-négociables

### 🛡️ RLS

- Le prof ne peut créer une visite **QUE pour ses élèves affectés**
- Côté serveur, vérifier via `teacher_assignments`
- Côté UI, ne pas afficher le bouton "Nouvelle visite" si pas affecté

### 🛡️ Mobile-first

- Sauvegarde brouillon **automatique** toutes les 10s (localStorage)
- Pour ne pas perdre 10 min de saisie en cas de coupure réseau
- Footer sticky avec boutons "Brouillon" et "Valider plus tard"
- Pas de champs obligatoires excessifs

### 🛡️ Audit log

Insérer dans `audit_logs` :
```
{
  user_id, establishment_id, action: 'visit_create',
  target_type: 'visit', target_id: <visit_id>,
  metadata: { student_id, type_contact, alerte_niveau }
}
```
