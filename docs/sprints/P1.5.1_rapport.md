# P1.5.1 - Rapport Codex - Roles et permissions pro

Date : 13 mai 2026  
Branche : `feature/p1-5-1-roles-permissions`  
Verdict : READY_FOR_AUDIT

## Objectif

Corriger le blocage metier sur les invitations DDFPT et stabiliser une hierarchie de roles coherente pour les pilotes lycee : admin, DDFPT, professeur principal, referent PFMP, tuteur et eleve.

## Diagnostic

Le blocage DDFPT venait de deux endroits :

- `src/server/invitations.functions.ts` interdisait a un admin d'inviter un `ddfpt`.
- `src/routes/admin.users.tsx` et `InviteUserForm` n'exposaient pas dynamiquement les roles selon le role connecte.

Ancienne logique :

```ts
if (caller.role === 'admin' && (data.role === 'admin' || data.role === 'ddfpt')) {
  throw new Error('Seul le superadmin peut inviter un admin ou un DDFPT.')
}
```

Nouvelle logique : `canInvite(caller.role, data.role)` depuis `src/lib/permissions.ts`.

## Notes techniques

- `students.referent_id` etait absent du schema local. Une migration idempotente a ete ajoutee, comme prevu par le brief.
- La source historique `ROLE_LABELS` dans `@/types` reste compatible, mais re-exporte maintenant la source `@/lib/permissions`.
- Le filtrage principal sur classes/eleves est applique cote UX. La RLS actuelle reste plus large sur `classes`; un durcissement SQL dedie pourra etre fait apres validation metier, pour eviter de casser le pilote.
- Les server functions P1.6 sont volontairement des stubs : elles valident l'input, verifient la session, puis retournent `Non implemente - sera livre en P1.6`.
