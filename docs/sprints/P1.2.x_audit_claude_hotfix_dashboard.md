## Hotfix appliqué — `fetchSetupChecklist` cassait le Dashboard

### Symptôme constaté en prod

Sur `https://www.pfmp-pilot.fr/dashboard`, à l'ouverture :

```
Impossible de charger le dashboard
fetchSetupChecklist establishment: column establishments.establishment_id does not exist
```

Le dashboard ne pouvait plus charger du tout, ce qui rendait la SetupChecklist (livrable principal de PR #25) **invisible en prod** — d'où l'impression de BraveHeart que "tous les boutons ne marchent pas" : aucun bouton ne pouvait fonctionner sur une page qui ne s'affichait pas.

### Cause racine

`src/services/dashboard.ts:160-172` (`fetchSetupChecklist`) utilisait la fonction utilitaire `applyEstablishmentScope(query, scope)` sur la table `establishments`.

`applyEstablishmentScope` ajoute systématiquement `.eq('establishment_id', scope)`. Cette colonne existe sur toutes les tables filles (`classes`, `students`, `pfmp_periods`, etc.) mais **pas sur `establishments` elle-même** : la PK de `establishments` est `id`. Postgres renvoie `column establishments.establishment_id does not exist`, l'erreur se propage jusqu'au composant Dashboard via `throw` et le `<EmptyState>` d'erreur est affiché.

### Fix appliqué

Commit `8f74631` sur main (push direct via injection CodeMirror sur github.com/edit) :

```ts
async function fetchSetupChecklist(scope?: string | null): Promise<DashboardSetupChecklist> {
  const sb = getSupabase()
  // NOTE: la table `establishments` n'a pas de colonne `establishment_id`. Sa PK est `id`.
  // On ne peut donc PAS utiliser applyEstablishmentScope ici : il faut filtrer par `id`.
  const establishmentBase = sb.from('establishments').select('city,uai').order('created_at').limit(1)
  const establishmentQuery = scope ? establishmentBase.eq('id', scope) : establishmentBase
  // ...
}
```

5 lignes modifiées, fonction isolée, pas d'impact sur les autres usages d'`applyEstablishmentScope` (qui restent corrects sur les tables filles).
