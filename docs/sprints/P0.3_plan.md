# P0.3 - Plan d'execution

## Contexte et objectif

P0.2.1 a corrige le blocker RLS/Auth : le profil superadmin reel se charge
et le switcher tenant apparait. P0.3 doit maintenant retirer le dashboard du
monde demo en prod.

Objectif : `src/routes/dashboard.tsx` garde le rendu demo uniquement quand
`isDemoMode() === true`. En prod Supabase, la page lit les tables reelles via
un service `src/services/dashboard.ts`, avec RLS naturelle, loading skeleton et
empty states propres si le tenant est vide.

Note de synchronisation : `SYNC.md` n'est pas encore actualise apres le merge
P0.2.1, mais BraveHeart a explicitement demande d'avancer vers P0.3.

## Fichiers prevus

- `src/services/dashboard.ts` : nouveau service Supabase pour les donnees du dashboard.
- `src/routes/dashboard.tsx` : separation claire `DashboardDemo` / `DashboardSupabase`.
- `docs/sprints/P0.3_rapport.md` : rapport final apres implementation.
- Optionnel : un petit composant local `DashboardSkeleton` dans `dashboard.tsx`
  si aucun skeleton global n'existe.

Aucun changement schema/RLS prevu.

## Architecture technique

Flux cible :

```text
dashboard route
  -> isDemoMode()
      -> DashboardDemo : code actuel conserve avec data/demo
      -> DashboardSupabase : useEffect + service
           -> src/services/dashboard.ts
              -> getSupabase()
              -> queries count/head + quelques fetchs agreges
```

Le service restera client-side pour P0.3 afin de respecter le cablage actuel
AuthProvider/session Supabase et eviter d'introduire un nouveau systeme de
server functions sur ce sprint.

## Requetes Supabase prevues

RLS filtre le tenant. Pas de `.eq('establishment_id', ...)`.

### `fetchCurrentPeriod()`

Table : `pfmp_periods`

```ts
from('pfmp_periods')
  .select('*')
  .eq('status', 'in_progress')
  .order('start_date', { ascending: false })
  .limit(1)
  .maybeSingle()
```

Fallback si aucune periode en cours : empty state principal "Aucune periode
PFMP en cours".

### `fetchDashboardKpis(periodId)`

Requetes count simples :

- `students` : count total eleves visibles
- `placements` avec `period_id = periodId` et status in
  `signed_convention,in_progress,completed` pour eleves en PFMP
- `placements` avec `period_id = periodId` et status `no_stage` pour eleves sans entreprise
- `visits` avec `period_id = periodId` et status `validated` pour visites realisees
- `alerts` avec `type = visit_late` et `resolved = false` pour visites en retard
- `documents` avec `period_id = periodId`, `type = convention`, `status = missing`
- `documents` avec `period_id = periodId`, `type = attestation`, `status = missing`

Calculs derives :

- `criticalMissing = conventionsMissing + attestationsMissing`
- taux affectation / visites / documents en regle avec protections division par zero.

### `fetchTeacherLoads(periodId)`

Fetch agrege JS acceptable en P0.3 :

- `teachers.select(id, first_name, last_name)`
- `teacher_assignments.select(teacher_id).eq('period_id', periodId)`

Aggregate par `teacher_id`, seuil local `TEACHER_OVERLOAD_THRESHOLD = 6`.

### `fetchCompanyNetwork(periodId)`

Fetch agrege JS acceptable :

- `companies.select(id, name, city, sector, professional_family, status, students_hosted)`
- `placements.select(company_id).eq('period_id', periodId)`

KPIs :

- entreprises actives sur la periode
- partenaires forts : `students_hosted >= 3`
- a relancer : status `to_recontact` ou `to_watch`
- familles couvertes et top secteurs

### `fetchAlerts(limit)`

Table : `alerts`

```ts
from('alerts')
  .select('*')
  .eq('resolved', false)
  .order('created_at', { ascending: false })
  .limit(limit)
```

Le composant `AlertList` attend les types applicatifs demo, donc P0.3 devra
mapper `AlertRow` vers le shape minimal attendu ou afficher une liste dashboard
dediee plus simple. Je privilegie une liste dediee locale pour eviter de
toucher `AlertList` hors scope.

### `fetchActivityLog(limit)`

Il n'existe pas encore de table `activityLog` applicative. Utiliser
`audit_logs` comme source reelle :

```ts
from('audit_logs')
  .select('*')
  .order('created_at', { ascending: false })
  .limit(limit)
```

Si aucune ligne : empty state "Aucune activite recente".

## Empty states prevus

- Aucun profil/session en prod : message "Session requise" + lien login si le
  composant se retrouve charge sans profil.
- Aucune periode PFMP : grand empty state "Aucune periode PFMP en cours" avec
  action vers `/pfmp-periods`.
- Aucune donnee eleve/placement : KPIs a 0, texte "Aucune donnee pour cette periode".
- Aucune alerte : "Aucune alerte active".
- Aucune activite : "Aucune activite recente".
- Aucun reseau entreprise : "Aucune entreprise renseignee".

## Demo intacte

Le code actuel de calcul depuis `@/data/demo` sera deplace dans un composant
`DashboardDemo`. Il doit rester fonctionnel quand `isDemoMode()` est vrai.

Point additionnel necessaire pour eviter le "non vendable" en prod :
`Sidebar` affiche toujours le role switcher demo. Dans P0.3, je prevois de le
masquer hors demo avec `isDemoMode()`. Cela touche `src/components/Sidebar.tsx`,
mais c'est directement lie au critere "prod ne doit plus afficher la demo".

## Loading

Pas de composant skeleton existant dans `src/components/ui`. Je prevois un
`DashboardSkeleton` local dans `dashboard.tsx` avec cartes grises stables,
sans ajouter de dependance.

## Ordre d'attaque

1. Creer `src/services/dashboard.ts` avec types de DTO dashboard et fonctions
   Supabase.
2. Refactorer `dashboard.tsx` en `DashboardPage`, `DashboardDemo`,
   `DashboardSupabase`, `DashboardSkeleton`, petits composants empty/listes.
3. Masquer le role switcher demo dans `Sidebar` quand `isDemoMode()` est faux.
4. Verifier mode demo par build/typecheck et inspection locale.
5. Verifier REST/service prod autant que possible.
6. Rediger `docs/sprints/P0.3_rapport.md`.

## Tests prevus

- `npm run typecheck`
- `npm run build`
- REST Supabase via Node si utile pour confirmer counts sur tenant vide.
- Smoke UI prod : si navigateur non disponible cote Codex, demander validation
  BraveHeart par captures comme en P0.2.1.

## Risques et limites

- La DB prod est quasiment vide : le rendu attendu post-P0.3 sera souvent un
  empty state, pas un dashboard riche.
- Les pages enfants (`students`, `companies`, `documents`, etc.) resteront demo
  jusqu'aux sprints P0.4+.
- Les donnees long terme "cursus 3 ans" et moteur documentaire etablissement
  sont structurantes, mais hors scope P0.3. Elles doivent rester dans la vision
  produit, pas etre codees en avance ici.
