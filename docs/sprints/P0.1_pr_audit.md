⚠️ **Point de clarification sur le SHA-256 :**

Tu as noté dans le rapport : *"le SHA-256 brut local de `src/data/demo.ts`
diffère du hash annoncé par Claude, probablement à cause des fins de
ligne/encodage Windows"*.

J'ai recalculé le SHA-256 dans mon environnement Linux ce matin sur le
fichier exact tel qu'il est dans le commit `pr-7` :

```bash
sha256sum src/data/demo.ts
e7f7f354d377c2317121733b8d0469e7992b7becc14497f2f66dc39e97440dab  src/data/demo.ts
```

C'est **exactement** le SHA que j'avais noté dans le commit `5b6831a`.
**Pas de divergence côté serveur.**

L'écart que tu as observé localement vient probablement de ton
environnement (CR/LF Windows ajoutés lors d'un `git checkout`, ou
core.autocrlf=true dans la config Git locale, ou bien un encodage de
relecture). Ce n'est **pas un blocker** pour P0.1 puisque ta
comparaison ligne à ligne après en-tête confirme l'intégrité du
contenu (méthode plus fiable qu'un SHA brut sur des fichiers texte
multi-plateforme).

→ Recommandation pour les futurs sprints : vérifie ton `git config
core.autocrlf` et ton `.gitattributes`. Pour un repo Linux/Mac/Windows
mixte, on veut typiquement `core.autocrlf=input` ou un
`.gitattributes` qui force `* text=auto eol=lf`. À traiter dans un
sprint d'hygiène plus tard, pas urgent.
