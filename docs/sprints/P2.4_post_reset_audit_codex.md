## Diagnostic

Le bug "Etablissement actif requis" n'etait pas lie au reset lui-meme. Il venait de l'incoherence suivante :

1. Le client lit `active_establishment_id` sans tenir compte de `active_establishment_expires_at`.
2. Le serveur, lui, refusait d'injecter `active_establishment_id` si `active_establishment_expires_at` etait expire.
3. Le switcher global `SuperadminTenantSwitcher` mettait a jour `active_establishment_id` sans renouveler `active_establishment_expires_at`.
4. Le bouton "Se connecter en tant que" ne donnait que 15 minutes, trop court pour un onboarding complet.
5. Apres suppression d'un etablissement, un ancien `active_establishment_id` pouvait rester dans le JWT et pointer vers un tenant inexistant.

Effet utilisateur : le header semblait encore avoir un lycee actif, mais certaines server functions voyaient un superadmin sans tenant actif et retournaient des erreurs comme :

- `Etablissement actif requis pour lire les professeurs.`
- `Etablissement actif requis pour gerer les professeurs.`
- `Etablissement appelant introuvable.`

## Correctif applique dans la branche

Branche : `feature/p2-4-tenant-context-hardening`

### `src/server/_lib.ts`

- Ajout validation UUID centralisee locale.
- Ajout validation expiration du scope actif.
- Ajout verification que l'etablissement actif existe encore et n'est pas inactif.
- Injection du tenant actif uniquement si :
  - role = `superadmin`,
  - profil sans `establishment_id`,
  - `active_establishment_id` est un UUID valide,
  - expiration absente ou non expiree,
  - etablissement existe toujours.

### `src/lib/auth.ts`

- `readActiveEstablishmentId()` respecte maintenant :
  - format UUID strict,
  - `active_establishment_expires_at`.
- Si le contexte est expire, l'UI ne pretend plus qu'un tenant est actif.

### `src/components/SuperadminTenantSwitcher.tsx`

- Le switcher global renouvelle aussi `active_establishment_expires_at`.
- Duree de travail : 8 heures.
- La sortie en "Vue globale" vide aussi l'expiration.

### `src/server/superadminDashboard.functions.ts`

- `startSuperadminImpersonation()` passe de 15 minutes a 8 heures.
- Cela couvre un onboarding complet sans recreer le bug pendant la session.
