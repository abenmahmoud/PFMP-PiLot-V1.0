# P1.2 — Invitations admin/DDFPT et cockpit utilisateurs tenant — Plan Codex

## Objectif fonctionnel

Permettre au superadmin BraveHeart de créer un établissement puis d'inviter son premier admin/DDFPT, et permettre ensuite à cet admin/DDFPT d'inviter les utilisateurs de son établissement depuis un cockpit tenant.

## Position dans la roadmap

P1.0 a livré la création d'établissement.
P1.1 a livré la fiche détail établissement, le tenant switcher superadmin et le domaine `pfmp-pilot.fr`.

P1.2 livre la première vraie brique SaaS B2B : l'onboarding d'utilisateurs par invitation, sans encore importer les données métier.

## Parcours cible

1. Le superadmin se connecte sur `www.pfmp-pilot.fr`.
2. Il crée un établissement.
3. Depuis la fiche établissement, il invite un admin ou DDFPT.
4. L'utilisateur reçoit un lien d'invitation Supabase.
5. Il arrive sur `/onboarding`, définit son mot de passe, puis accède à son dashboard tenant vide.
6. L'admin/DDFPT peut ouvrir `/admin/users` et inviter des profils `principal`, `referent` ou `eleve`.
7. Le superadmin garde la main via `/superadmin/establishments/$id` et le tenant switcher.

## Contraintes non négociables

- Aucun `service_role` côté client.
- Les invitations passent par une fonction serveur TanStack Start, exécutée côté Vercel.
- Le serveur revalide le JWT utilisateur avant chaque invitation.
- Superadmin peut inviter `admin`, `ddfpt`, `principal`, `referent`, `eleve`.
- Admin/DDFPT tenant peut inviter uniquement dans son établissement et pas de rôle `admin`/`ddfpt`.
- Les données métier restent protégées par RLS ; la fonction serveur ne sert qu'à Supabase Auth Admin.
- Mode démo préservé : les nouvelles routes affichent un état explicite ou restent non disponibles en démo.
- Build + typecheck verts.

## Décisions techniques

### Pourquoi TanStack server function et pas Edge Function Supabase ?

Pour P1.2, l'app est déjà déployée sur Vercel via TanStack Start. Une fonction serveur permet de livrer vite, sans ajouter un cycle de déploiement Supabase Functions. Les secrets restent côté serveur Vercel.

Supabase Edge Functions restent une option de durcissement plus tard, si on veut tout centraliser côté Supabase.

### Pourquoi pas de table `invitations` maintenant ?

Le MVP peut fonctionner avec `auth.users` + `profiles` + `audit_logs`. Une table dédiée sera utile plus tard pour afficher des statuts détaillés (`pending`, `accepted`, `expired`), mais ce n'est pas bloquant pour l'onboarding réel.

## Critères d'acceptation

- [ ] Superadmin peut inviter un `admin` ou `ddfpt` depuis la fiche établissement.
- [ ] Admin/DDFPT peut inviter `principal`, `referent`, `eleve` depuis `/admin/users`.
- [ ] Admin/DDFPT ne peut pas inviter `admin`/`ddfpt`.
- [ ] Non-admin ne peut pas appeler la fonction serveur utilement.
- [ ] Les profils créés ont le bon `establishment_id`.
- [ ] `/onboarding` permet de définir un mot de passe.
- [ ] `/dashboard` tenant vide reste propre après onboarding.
- [ ] `audit_logs` reçoit `user.invited`.
- [ ] `npm run typecheck` OK.
- [ ] `npm run build` OK.
