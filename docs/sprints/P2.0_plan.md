# P2.0 — Dashboard Superadmin evolue

## Objectif

Livrer une console superadmin lisible et commerciale pour piloter plusieurs etablissements depuis un cockpit unique : KPIs groupe, taux de placement, etablissements a risque, entreprises mutualisees, activite cross-tenant et sante systeme.

## Etat de depart

- P1.9 est merge dans `main` avec periodes PFMP, placements et matchmaking.
- P1.7 a deja separe les portails `/admin`, `/prof`, `/eleve` et pose une base `/superadmin`.
- La route `/superadmin/` existe mais repose encore majoritairement sur un service client `src/services/superadmin.ts`.
- La sidebar admin contient deja une section superadmin, mais elle pointe encore vers `/superadmin` comme vue globale.

## Decisions techniques prevues

- Les aggregations seront d'abord faites via SELECTs service-role bornes et aggregation TypeScript, avec cache 60s, car aucune vue SQL/RPC dediee n'existe encore. Le rapport documentera le passage futur vers vues SQL/materialized views pour 50+ etablissements.
- L'impersonation P2.0 sera une "prise de contexte tenant" auditee pour le debug superadmin, pas une vraie session Supabase d'un autre utilisateur. C'est plus sur et suffisant pour le MVP commercial.
- Recharts n'est pas present dans `package.json`; le projet utilise deja `chart.js` + `react-chartjs-2`. J'utiliserai donc l'existant pour eviter une dependance supplementaire non justifiee, sauf si le code prouve entre-temps que Recharts est deja disponible.

## Risques surveilles

- Seroval : tous les retours server functions passent par `safeHandlerCall`.
- Performance : les fonctions de dashboard ont un cache 60s et selectionnent uniquement les colonnes utiles.
- Routes TanStack : conserver les parents avec `<Outlet />` et eviter les collisions avec les routes superadmin deja existantes.
- Cross-tenant : aucune fonction superadmin ne doit repondre sans profil `superadmin`.
