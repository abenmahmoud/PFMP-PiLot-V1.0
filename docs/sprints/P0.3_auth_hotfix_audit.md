## Audit du diff PR #13

### `src/lib/auth.ts`

Avant PR #13, le callback Supabase etait async :

```ts
supabase.auth.onAuthStateChange(async (_event, session) => {
  const profile = await fetchProfile(session.user.id)
  callback(...)
})
```

Apres PR #13 :

```ts
supabase.auth.onAuthStateChange((_event, session) => {
  window.setTimeout(async () => {
    const profile = await fetchProfile(session.user.id)
    callback(...)
  }, 0)
})
```

Ce changement est correct.

La documentation officielle Supabase indique que le callback
`onAuthStateChange` peut creer un deadlock si on `await` un autre appel
Supabase dedans, et recommande de deferer ces appels avec `setTimeout(..., 0)`.
Reference : https://supabase.com/docs/client/auth-onauthstatechange

Conclusion : la cause technique suspectee par Codex est credible et le patch
applique le contournement officiel.

### `src/lib/AuthProvider.tsx`

PR #13 ajoute :

- `AUTH_PROVIDER_TIMEOUT_MS = 10000`
- un `window.setTimeout` qui force `loading=false` avec l'erreur
  `Resolution de session Supabase trop longue.`
- un `clearTimeout` quand `buildAuthState()` ou `subscribeToAuthChanges()`
  resout l'etat.

Ce filet de securite ne corrige pas le deadlock en lui-meme, mais il empeche
l'app de rester indefiniment en skeleton. C'est une mitigation saine.

### Interaction avec `/dashboard`

`src/routes/dashboard.tsx` garde son propre timeout UI :

- apres 8 secondes : affiche "Session en attente" ;
- si `AuthProvider` finit ou timeoute a 10 secondes : `auth.loading` passe a
  `false`, donc le dashboard sort de l'etat "Session en attente".

Donc, apres PR #13, l'ecran "Session en attente" peut encore apparaitre
temporairement si l'auth met plus de 8 secondes, mais il ne doit plus rester
bloque indefiniment.

## Cause racine

Cause racine confirmee pour le bug cible :

1. `AuthProvider` subscribe a `supabase.auth.onAuthStateChange`.
2. Avant PR #13, le callback appelait `await fetchProfile()`.
3. `fetchProfile()` utilise le meme client Supabase pour lire `profiles`.
4. Supabase documente que ce pattern peut deadlocker le client Auth.
5. Le dashboard attend `auth.loading=false`.
6. L'auth reste en chargement, donc le dashboard affiche "Session en attente".

PR #13 casse cette boucle en deferant `fetchProfile()` hors du callback
synchrone de Supabase Auth.

## Risques residuels non bloquants

1. `buildAuthState()` appelle toujours `getCurrentSession()` puis
   `fetchProfile()` sans `Promise.race` interne. `AuthProvider` borne le rendu
   UI a 10 secondes, donc ce n'est pas un blocage utilisateur infini, mais si
   un navigateur continue a tomber sur "Session requise" apres 10 secondes,
   il faudra ajouter un timeout interne autour de `getCurrentSession()` et
   `fetchProfile()`.

2. Le message `Resolution de session Supabase trop longue.` devrait etre
   corrige en `Resolution` accentuee ou reformule proprement en francais dans
   un micro-commit UX. Non bloquant.

3. `SYNC.md` est fortement obsolète et peut induire les agents en erreur. Ce
   n'est pas lie a PR #13, mais il faut le remettre a jour rapidement.

4. Le test navigateur connecte reste a refaire avec Chrome MCP ou manuellement :
   console, network Supabase, localStorage `sb-...-auth-token`, refresh F5.
