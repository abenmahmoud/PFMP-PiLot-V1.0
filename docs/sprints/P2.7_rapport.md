# P2.7 — Rapport Document Intelligence PFMP

## Resultat

Le studio documentaire passe d'un simple modele manuel a une base durable pour importer les documents reels des lycees : DOCX, PDF remplissable, PDF plat ou scan. Le systeme cree un modele, archive la source si le bucket est disponible, detecte les champs essentiels et force une revue humaine avant affectation a une classe.

## Decisions techniques

- Ajout de `document_template_fields` pour separer clairement le modele source du mapping exploitable.
- Extension de `document_templates` avec `source_storage_path`, `source_mime_type`, `analysis_status`, `field_count` et `requires_human_review`.
- Analyse hybride :
  - Claude si `ANTHROPIC_API_KEY` et texte source disponible ;
  - heuristique PFMP officielle sinon, pour ne jamais bloquer le pilote.
- Upload Storage en best-effort dans `DOCUMENT_TEMPLATE_SOURCE_BUCKET` (`documents-private` par defaut). L'analyse reste utilisable meme si le bucket n'est pas encore cree.
- UI sans bouton mort : le panneau "Import intelligent" cree vraiment le modele et affiche les champs detectes.

## Limites assumées

- Le parsing natif DOCX/PDF/OCR complet n'est pas encore inclus pour eviter une dependance lourde dans ce sprint. L'utilisateur peut ajouter le fichier source et coller du texte/OCR si besoin ; le stockage et le mapping sont deja prets.
- La generation PDF finale et la signature multi-signataires restent le sprint suivant, branchees sur ce mapping.
