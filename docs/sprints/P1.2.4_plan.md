## 4. Lot 1 — Migration SQL + types

### Fichiers prevus

- `supabase/migrations/20260512120000_add_archived_at_to_profiles.sql`
- `src/lib/database.types.ts`

### Migration

Ajouter :

- `profiles.archived_at timestamptz null`
- index partiel `idx_profiles_archived_at`
- commentaire colonne

Je verifierai aussi si une policy RLS doit explicitement filtrer les comptes archives. Ma preference pour ce sprint :

- ne pas modifier lourdement les policies existantes si non necessaire ;
- filtrer les archives dans les server functions/listes par defaut ;
- laisser les superadmins voir les archives via la route dediee ;
- documenter cette decision dans le rapport.

Raison : la suppression logique touche la gestion operationnelle ; modifier les policies `profiles` peut avoir un effet transversal sur auth, onboarding et `fetchProfile`. Si un filtre RLS global cache le profil d'un utilisateur archive, il peut casser le chargement auth d'une maniere difficile a diagnostiquer. Je privilegie donc un filtrage applicatif explicite dans ce sprint, sauf preuve contraire pendant les tests.

### Types

Ajouter `archived_at: string | null` a `ProfileRow`.

Creer aussi un type local enrichi dans `src/server/users.functions.ts` ou `src/types/users.ts` :

```ts
export interface UserRowEnriched extends ProfileRow {
  email_confirmed_at: string | null
  last_sign_in_at: string | null
  establishment_name: string | null
  establishment_slug: string | null
}
```

Je privilegie `src/server/users.functions.ts` au depart pour garder le scope court. Si le type est partage par plusieurs composants, je le sortirai dans `src/types/users.ts`.

### Lecture `auth.users`

Supabase Admin API ne fournit pas une jointure SQL directe `profiles` -> `auth.users` cote frontend. Donc les fonctions server-side feront :

1. lecture `profiles` via admin client ;
2. lecture Auth Admin via `adminClient.auth.admin.listUsers()` ;
3. enrichissement en memoire par `id`.

Pour limiter le risque performance :

- `/admin/users` : nombre faible par tenant, enrichissement simple ;
- `/superadmin/users` : MVP avec pagination logique possible plus tard, mais pour l'etat actuel de la base le volume est faible.

Si `listUsers()` expose pagination dans cette version Supabase, je limiterai a une taille raisonnable et documenterai la suite.

### Permissions

Je vais coder les permissions cote serveur, sans confiance dans l'UI.

Regles :

- superadmin : toutes actions sauf self-delete et dernier superadmin actif ;
- admin tenant : actions uniquement sur son `establishment_id` ;
- admin tenant ne peut pas toucher `admin` ou `superadmin` ;
- admin tenant ne peut pas promouvoir vers `admin` ou `superadmin` ;
- ddfpt : lecture seulement selon brief ;
- principal/referent/eleve/tuteur : refus.

Point d'attention : le brief dit que les admins peuvent inviter, mais pas les DDFPT. Le code existant permet actuellement `admin` et `ddfpt` d'inviter. Je ne casserai pas `inviteUserToEstablishment` dans ce sprint, mais les nouvelles actions P1.2.4 respecteront le tableau du brief. Je documenterai cette divergence historique dans le rapport.

### Actions sensibles

#### Resend invitation

Je verifierai le comportement Supabase :

- si `inviteUserByEmail(email, ...)` fonctionne pour un email deja invite/pending, je l'utiliserai ;
- sinon fallback vers `generateLink({ type: 'invite', email })` si compatible.

Dans tous les cas :

- refuser si `email_confirmed_at !== null` ;
- audit log `user.invitation.resent`.

#### Reset password

- seulement pour user confirme ;
- `adminClient.auth.admin.generateLink({ type: 'recovery', email })` si possible ;
- si l'API ne declenche pas l'envoi email automatiquement, je documenterai le blocage et garderai la fonction en erreur explicite plutot que de faire semblant.

#### Update email

- pending only ;
- `auth.admin.updateUserById(userId, { email: newEmail, email_confirm: false })` ;
- update `profiles.email` ;
- audit log ancien/nouveau email.

#### Change role

- update `profiles.role` ;
- synchroniser `user_roles` :
  - supprimer les roles existants du meme `establishment_id` ;
  - inserer le nouveau role si `establishment_id` non null ;
- pour `referent`/`principal`, envisager de creer/mettre a jour une ligne `teachers` comme dans `invitations.functions.ts` si necessaire. Si je reutilise le helper, je le factoriserai prudemment ou le dupliquerai localement pour ne pas modifier l'API existante.

#### Change establishment

- superadmin only ;
- update `profiles.establishment_id` ;
- synchroniser `user_roles` ;
- si user devient `superadmin`, `establishment_id` peut rester null ;
- audit log.

#### Soft delete / restore

- `profiles.archived_at = now()` / `null` ;
- ne touche pas `auth.users` ;
- audit log.

#### Hard delete

- superadmin only ;
- pas self-delete ;
- `confirmEmail` doit matcher exactement ;
- dernier superadmin actif impossible ;
- audit log avant suppression avec snapshot ;
- suppression dans l'ordre prudent :
  1. `audit_logs where user_id = target`
  2. `user_roles where user_id = target`
  3. `profiles where id = target`
  4. `auth.admin.deleteUser(target)`

Point a verifier pendant code : si `profiles.id references auth.users(id) on delete cascade`, supprimer `auth.users` pourrait cascader `profiles`, mais je suivrai l'ordre du brief pour eviter FK surprises.

## 11. Risques et points de vigilance

### Supabase Auth resend/recovery

Les methodes Admin Auth peuvent varier selon la version SDK. Je verifierai pendant implementation :

- si `inviteUserByEmail` renvoie vraiment un email pour un pending existant ;
- si `generateLink({ type: 'recovery' })` envoie l'email ou genere seulement un lien.

Je ne coderai pas une fausse reussite : si l'API genere seulement un lien sans email, le message UI sera explicite ou la fonction sera ajustee selon l'API disponible.

### `auth.users` pagination

`listUsers()` peut etre pagine. Pour le MVP, le volume est faible. Je documenterai la limite et prevois une fonction extensible.

### RLS `archived_at`

Je ne veux pas casser `fetchProfile` en cachant le profil archive au mauvais endroit. Mon plan est de filtrer les archives dans les listes et actions, pas dans une policy globale, sauf necessite prouvee.

### DDFPT invite deja autorise historiquement

Le code existant autorise DDFPT a inviter. Le brief P1.2.4 dit DDFPT lecture seule. Je n'introduis pas de breaking change sur `inviteUserToEstablishment`, mais les nouvelles actions suivent le brief. Je signalerai cette divergence a BraveHeart/Claude dans le rapport.

### Encodage docs

Plusieurs docs affichees dans PowerShell sortent avec mojibake, mais les fichiers source sont utilises tels quels. Je garderai les nouveaux docs en UTF-8.
