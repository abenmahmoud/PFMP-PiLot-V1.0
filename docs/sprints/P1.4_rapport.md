## Migration SQL

Table ajoutee : `public.student_access_codes`

Colonnes principales :

- `establishment_id`
- `student_id`
- `code_hash`
- `code_hint`
- `status` (`active`, `revoked`, `expired`)
- `expires_at`
- `last_used_at`
- `created_by`
- `revoked_at`

Index et contraintes :

- un seul code actif par eleve : unique index partiel sur `student_id where status='active'`
- pas deux codes actifs identiques : unique index partiel sur `code_hash where status='active'`
- index tenant, student, status

RLS :

- superadmin : lecture/ecriture
- admin/DDFPT : lecture/ecriture dans son tenant
- principal : lecture/ecriture uniquement sur les eleves de ses classes (`classes.principal_id = auth.uid()`)
- delete : superadmin uniquement

## Server functions

Fichier : `src/server/studentAccessCodes.functions.ts`

Exports :

- `listClassStudentAccess(input)` : classe + eleves + statut code
- `generateClassStudentAccessCodes(input)` : generation manquants ou regeneration totale
- `generateSingleStudentAccessCode(input)` : regeneration d'un eleve
- `revokeStudentAccessCode(input)` : revocation du code actif d'un eleve

Securite :

- `auth.getUser(accessToken)` au debut de chaque action
- lecture du profil appelant via service-role
- permissions verifiees manuellement avant mutation
- service-role jamais expose au client
- audit log obligatoire sur generation classe, generation individuelle et revocation

## Strategie code clair / hash

Format code clair :

```text
PFMP-XXXX-XXXX
```

Alphabet sans caracteres ambigus : `ABCDEFGHJKLMNPQRSTUVWXYZ23456789`.

Le code clair est retourne uniquement dans la reponse de generation/regeneration pour permettre l'impression immediate. Il n'est jamais stocke.

Stockage :

- `code_hash = sha256(STUDENT_ACCESS_CODE_PEPPER + code_normalise)`
- si `STUDENT_ACCESS_CODE_PEPPER` est absent, fallback sur `SUPABASE_SERVICE_ROLE_KEY`
- `code_hint` conserve uniquement les 4 derniers caracteres pour l'UI

Recommandation prod : ajouter `STUDENT_ACCESS_CODE_PEPPER` dans Vercel + environnement local, puis redeployer.

## UI livree

### `/classes`

Chaque carte classe a maintenant deux actions :

- `Codes eleves` -> `/classes/$id`
- `Voir les eleves` -> `/students`

### `/classes/$id`

Vue classe complete :

- stats eleves / codes actifs / codes manquants
- generation des codes manquants
- regeneration de toute la classe avec confirmation
- table eleves avec statut code
- action par eleve : regenerer, revoquer
- fiche imprimable avec code clair + QR

Le QR pointe vers :

```text
https://www.pfmp-pilot.fr/eleve?code=<CODE>
```

### `/eleve`

Portail eleve minimal :

- champ code personnel
- pre-remplissage via `?code=...`
- message clair : verification du code au sprint suivant

La connexion eleve effective reste hors perimetre P1.4.

## Tests manuels post-merge

1. Appliquer la migration Supabase.
2. Se connecter en superadmin/admin.
3. Importer le seed SIECLE si la base est vide.
4. Aller sur `/classes`.
5. Ouvrir `Codes eleves`.
6. Cliquer `Generer les manquants`.
7. Verifier la fiche imprimable + QR.
8. Recharger la page : les codes clairs ont disparu, seuls statut + hint restent.
9. Regenerer un eleve.
10. Revoquer un eleve.
11. Verifier `audit_logs` :
    - `student_access_codes.class_generated`
    - `student_access_codes.student_generated`
    - `student_access_codes.revoked`
