## Server functions ajoutees

Dans `src/server/superadminDashboard.functions.ts` :

- `getGlobalKpis({ accessToken })`
- `getEstablishmentBreakdown({ accessToken })`
- `getCrossEstablishmentInsights({ accessToken })`
- `getActivityFeed({ accessToken, limit, establishmentId, action })`
- `getMonthlyTrends({ accessToken })`
- `getSuperadminEstablishmentDetail({ accessToken, establishmentId })`
- `setEstablishmentActive({ accessToken, establishmentId, active })`
- `startSuperadminImpersonation({ accessToken, establishmentId })`
- `getSystemHealth({ accessToken })`

Toutes les fonctions :

- utilisent `createAdminClient()` service-role cote serveur ;
- verifient `getCallerProfile()` puis `caller.role === 'superadmin'` ;
- passent par `safeHandlerCall()` pour eviter les problemes Seroval ;
- sanitizent les retours via le pattern `_lib.ts`.

## Decisions techniques

1. **Cache 60s**
   - Snapshot multi-table en memoire module avec TTL 60s.
   - Les mutations superadmin (`setEstablishmentActive`) purgent le cache.

2. **Aggregations TypeScript bornees**
   - Les aggregations sont faites cote serveur apres SELECT service-role avec colonnes metier existantes.
   - Pour 50+ etablissements, recommandation future : vues SQL/materialized views dediees.

3. **Graphique sans nouvelle dependance**
   - Le brief citait Recharts, mais `package.json` contient `chart.js/react-chartjs-2`, pas Recharts.
   - Pour eviter une dependance non necessaire, `PlacementTrendChart` utilise un SVG React leger.

4. **Impersonation securisee**
   - Pas de vol de session ni mot de passe.
   - P2.0 ouvre un contexte tenant superadmin via `active_establishment_id`, expire a 15 min et journalise `superadmin.impersonation.started`.
