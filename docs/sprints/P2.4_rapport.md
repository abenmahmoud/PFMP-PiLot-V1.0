# P2.4 - Attestations auto + signature electronique

Date : 14 mai 2026  
Branche : `feature/p2-4-signatures-electroniques`  
Verdict Codex : `READY_FOR_AUDIT`

## 1. Resume

P2.4 ajoute le socle complet de signature electronique simple pour les documents generes :

- demande de signatures depuis une attestation/document genere ;
- signature authentifiee pour referent/admin ;
- signature publique par magic link pour tuteur sans compte ;
- preuve cryptographique : hash SHA-256, date, IP, user-agent, methode ;
- finalisation PDF avec page de preuve et QR de verification ;
- tableau de bord admin des signatures ;
- mode batch par `batch_id` pour demandes et relances en masse ;
- parcours public `/sign/$token` et verification publique `/verify/$documentId`.

Le sprint a ete adapte a l'etat reel du schema : `generated_documents`, `documents`, `document_signatures` et `tutor_access_tokens` existaient deja partiellement. La migration P2.4 aligne donc le schema sans casser l'existant.

## 6. Securite et conformite

- Tokens magic link : le token clair n'est jamais stocke en DB. Seul `SHA-256(token)` est conserve.
- Token : generation aleatoire 32 bytes minimum via Web Crypto / fallback navigateur.
- Magic link : expiration obligatoire via `SIGNATURE_MAGIC_LINK_TTL_DAYS`.
- Signature publique : la route `/sign/$token` ne retourne jamais `magic_link_token_hash`, `code_hash` ou donnee interne.
- Preuve : chaque signature stocke `document_hash`, `signed_at`, IP, user-agent, methode.
- Immutable : trigger SQL bloque update/delete sur une signature `signed`.
- Audit logs : `signature.requested`, `signature.signed`, `signature.signed_magic_link`, `signature.reminder_sent`, `signature.revoked`, `signature.finalized`, `signature.batch_requested`, `signature.batch_reminders_sent`.
- Service-role : utilise uniquement dans les server functions.
- RGPD : modal de consentement avant signature publique + document `docs/legal/RGPD_signatures.md`.
- Mode demo : signatures factices locales, aucun email reel, aucune ecriture Supabase.

## 7. PDF final signe

`src/lib/pdfSignatures.ts` utilise `pdf-lib` et `qrcode` pour :

- charger le PDF original depuis le bucket `GENERATED_PDFS_BUCKET` ;
- ajouter une page finale "Dossier de preuve" ;
- integrer liste des signataires, dates, IP, user-agent, hash, URL de verification ;
- integrer un QR code vers `/verify/$documentId` ;
- uploader une version `signed-vN.pdf` ;
- stocker `final_signed_pdf_url`, `final_signed_sha256_hex` et `signature_proof`.

Si le PDF source n'est pas encore present dans Storage, la finalisation conserve quand meme une preuve JSON et un hash de preuve pour ne pas bloquer l'etat `fully_signed`.
