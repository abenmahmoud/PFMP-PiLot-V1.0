## Objectif livre

Ce hotfix corrige le blocage de navigation vers `/classes/$id` apres clic sur "Codes eleves" depuis `/classes`, puis ajoute sur `/students/$id` une card "Acces eleve" qui permet de voir l'etat du code actif et de generer, regenerer ou revoquer le code d'un eleve sans repasser par la fiche classe.

Aucune migration SQL n'a ete ajoutee. Le sprint reutilise la table `student_access_codes` et les server functions livrees en P1.4.

## Bug corrige : route `/classes/$id`

Cause racine :

`src/routes/classes.tsx` est traite comme route parente de `src/routes/classes.$id.tsx` par TanStack Router. Comme `ClassesPage` ne rendait pas `<Outlet />`, la route enfant etait bien matchee mais son composant ne s'affichait jamais. L'utilisateur restait donc visuellement sur la liste des classes.

Correction :

- ajout de `Outlet` et `useMatchRoute`
- detection de la sous-route `/classes/$id`
- rendu de `<Outlet />` uniquement quand la route enfant est active
- aucun changement sur `ClassCard`, `ClassLinks` ou `ClassesSupabase`

Effet attendu :

`/classes` continue d'afficher la liste. `/classes/$id` affiche maintenant la fiche classe avec le titre "Classe", le sous-titre "Codes eleves et acces sans email" et les actions de generation de codes.

## Feature ajoutee : card "Acces eleve"

`fetchStudentById()` recupere maintenant le code actif de l'eleve dans `student_access_codes` :

- `id`
- `code_hint`
- `status`
- `created_at`

L'erreur eventuelle sur cette lecture est volontairement non bloquante via `console.warn`, afin de ne jamais casser une fiche eleve si aucun code n'existe ou si la policy RLS ne retourne rien.

Dans `/students/$id`, la nouvelle card "Acces eleve" :

- affiche `Aucun code` si aucun code actif n'existe
- affiche `Actif ...XXXX` si un code actif existe
- affiche le code complet et le QR code uniquement apres generation ou regeneration, via l'etat local `freshCode`
- ne stocke jamais le code complet dans le detail permanent
- propose `Generer un code`, `Regenerer`, `Revoquer`
- renvoie vers `/classes/$id` avec "Voir toute la classe"

Le mode demo affiche la meme card avec `accessCode: null` et les actions desactivees.

## Hotfix Seroval

`listClassStudentAccess` serialise son resultat via TanStack Start/Seroval. Pour eviter un crash runtime si une row Supabase contient une propriete `undefined`, `buildClassStudentAccessResult()` passe maintenant son objet de retour dans `sanitizeForSeroval()`.

Cette fonction transforme uniquement les `undefined` en `null` via `JSON.stringify`/`JSON.parse`, ce qui garde le contrat JSON-compatible attendu par les server functions.

## Securite

- Aucun code complet n'est lu depuis Supabase.
- Seul `code_hint` est affiche depuis le stockage permanent.
- Le code complet n'est visible qu'une fois, dans la reponse de `generateSingleStudentAccessCode`.
- La generation et la revocation reutilisent les server functions P1.4, qui portent les controles serveur et la service-role key.
- Aucune nouvelle server function.
- Aucune migration SQL.
- Les retours `listClassStudentAccess` sont sanitises avant Seroval.
