# P2.4 - Plan d'attaque - Attestations auto + signature electronique simple

## Objectif

Finir le cycle PFMP cote documents : generer une attestation finale depuis un placement termine, demander les signatures, permettre la signature simple par utilisateur connecte ou tuteur via lien public, horodater/hash, puis fournir un statut de preuve et une verification publique.

## Etat reel du repo avant code

- `document_templates`, `documents`, `generated_documents`, `document_signatures` et `tutor_access_tokens` existent deja dans `0001_init.sql`.
- `document_signatures` a un modele ancien : `document_id`, `generated_document_id`, `signer_email`, `signer_role`, `status`, `signature_data`, `signed_document_sha256`, `ip_address`, `user_agent`.
- `generated_documents` contient `storage_path`, `sha256_hex`, `rendered_with`, mais pas encore `signature_status`, `required_signers`, `final_signed_pdf_url`.
- `/admin/documents` existe mais reste liste simple. Il n'y a pas encore `/admin/documents/$id`.
- `qrcode.react` est deja installe. `pdf-lib` et `signature_pad` ne sont pas installes.

## Decision d'architecture

1. Migration additive, pas de table signature recreate.
   - Ajouter les colonnes P2.4 manquantes a `document_signatures`, `generated_documents`, `tutor_access_tokens` si utile.
   - Creer `signature_request_emails`.
   - Garder compatibilite avec `status signature_status` existant.

2. Signature simple, pas signature qualifiee.
   - Texte UI clair : signature electronique simple, non qualifiee eIDAS.
   - Preuve : hash SHA-256 + timestamp + IP + user-agent + token usage.

3. Pas de dependance lourde pour MVP.
   - Canvas maison pour dessiner la signature au lieu de `signature_pad`.
   - QR verification via `qrcode.react` deja present.
   - PDF final : pour ce sprint, produire une preuve JSON/HTML et preparer `pdfSignatures.ts`; l'integration PDF binaire complete dependra de `pdf-lib` si la dependance est ajoutee ensuite.

4. Server functions strictes.
   - `safeHandlerCall` partout.
   - Service-role cote serveur seulement.
   - Audit log sur demande, signature, relance, revocation, finalisation.

5. Magic link tuteur compatible avec l'existant.
   - Utiliser `tutor_access_tokens` pour stocker le hash du token.
   - Le token clair n'est visible que dans l'URL envoyee.
   - `document_signatures.magic_link_token` du brief devient une colonne d'alignement optionnelle, mais le stockage securise reste `token_hash`.

## Ordre d'implementation

1. Migration SQL P2.4 additive.
2. Types DB : `GeneratedDocumentRow`, `DocumentSignatureRow`, `TutorAccessTokenRow`, `SignatureRequestEmailRow`.
3. Libs : `signatureCrypto.ts`, `pdfSignatures.ts`, `emailSignatureRequest.ts`.
4. Server functions : `signatures.functions.ts`, `signaturesBatch.functions.ts`.
5. Composants : `SignatureCanvas`, timeline, modal demande signatures, QR verification, RGPD consent.
6. Routes publiques : `/sign/$token`, `/verify/$documentId`.
7. Routes admin : `/admin/documents/$id`, `/admin/signatures`, sidebar.
8. Templates email HTML + docs RGPD.
9. Typecheck, build, rapport.

## Garde-fous

- Ne jamais presenter cette signature comme qualifiee eIDAS.
- Ne jamais stocker le token magique en clair comme unique preuve ; stocker son hash.
- Ne jamais autoriser UPDATE/DELETE de signature signee cote UI/server.
- Mode demo : pas d'email reel.
