## Statut

Blocker technique avant exécution complète du sprint.

## Contexte

Le plan `docs/sprints/P0.1_plan.md` a été committé avant toute modification applicative, conformément à la méthode collaborative.

## Blocker

Le brief demande une copie conforme de `src/data/demo.ts` dans :

```txt
data-reference/demo.reference.ts
```

avec seulement l’en-tête ajouté.

Dans l’environnement actuel, l’accès GitHub disponible tronque le fichier `src/data/demo.ts` : le fichier fait environ 597 lignes, mais le contenu récupérable par le connecteur est tronqué autour de la ligne 301. Je ne peux donc pas garantir une copie conforme intégrale du fichier.

Créer un fichier incomplet, une réexportation vers `src/data/demo.ts`, ou une copie partielle violerait le brief, notamment :

- `Pas de modification du contenu, juste l'en-tête ajouté.`
- `NE PAS IMPORTER ce fichier dans le code applicatif.`
- `Il existe uniquement comme spécification produit.`

## Proposition

Deux options propres :

### Option A — exécution locale par Claude/Codex avec repo complet

L’agent qui a accès au repo complet localement exécute :

```bash
mkdir -p data-reference
cat > data-reference/demo.reference.ts <<'EOF'
/**
 * ============================================================================
 * COPIE FIGÉE DE LA DÉMO — RÉFÉRENCE PRODUIT
 * ============================================================================
 *
 * Ce fichier est une copie figée de src/data/demo.ts au 7 mai 2026.
 * Il sert de référence visuelle : "voilà ce que la prod doit reproduire".
 *
 * ⚠️ NE PAS IMPORTER ce fichier dans le code applicatif.
 *    Il existe uniquement comme spécification produit.
 *    Pour la démo runtime → src/data/demo.ts
 *    Pour la prod runtime → hooks Supabase
 *
 * ============================================================================
 */
EOF
cat src/data/demo.ts >> data-reference/demo.reference.ts
```

Puis commit séparé :

```bash
git add data-reference/demo.reference.ts
git commit -m "docs: add frozen demo reference"
```

### Option B — autoriser une référence non conforme

Autoriser explicitement une référence légère, par exemple un fichier Markdown qui pointe vers le SHA de `src/data/demo.ts`, mais ce serait une déviation du brief P0.1.

Je ne recommande pas cette option sans validation.

## État des autres vérifications déjà faites

### `useCurrentUser`

Le fichier existe et bridge déjà partiellement :

- mode démo : utilisateur démo via `localStorage` ;
- mode prod avec `auth.profile` : conversion `ProfileRow` → `Profile` ;
- mode prod sans session/profile : fallback démo actuel.

Point à discuter : le brief demande `null/redirect vers /login` en prod sans session. Le code actuel garde un fallback démo si `!auth.profile`. Corriger cela proprement implique de changer le contrat de `useCurrentUser()` ou d’ajouter un guard d’auth plus large, car plusieurs composants attendent aujourd’hui un `Profile` non-null.

### `EmptyState`

Le composant existe déjà dans `src/components/EmptyState.tsx`.

API actuelle :

```tsx
<EmptyState
  icon={...}
  title="..."
  description="..."
  action={ReactNode}
/>
```

Le brief propose `action={{ label, href }}`. L’API actuelle est plus flexible et déjà utilisée par `RoleGuard`. Je propose de ne pas la modifier dans P0.1 sauf instruction contraire.
