## Decisions techniques prises

- `SetupChecklist` retourne `null` quand les 5 etapes sont terminees. Le mode demo lui passe donc 5 etapes `done: true`, ce qui evite toute regression visuelle sur Jean Moulin.
- Les donnees de checklist sont integrees a `fetchDashboardData()` au lieu d'ajouter un fetch dedie cote route. Le dashboard a ainsi une seule source de verite pour l'etat de setup.
- Le calcul respecte le brief : 1 select identite sur `establishments` + 3 count queries (`classes`, `students`, `pfmp_periods`).
- Le save `/settings` utilise `Promise.allSettled` pour distinguer l'echec identite et l'echec parametres. Aucun message "saved" n'est affiche si l'une des deux operations echoue.
- Le service `updateTenantSettings()` retourne maintenant `{ establishment, settings }`, ce qui permet de synchroniser l'etat local apres sauvegarde sans reload.

## Empty / done states geres

- Checklist visible tant qu'au moins une etape reste a faire.
- Checklist masquee automatiquement a 5/5.
- Etape compte toujours validee pour un utilisateur authentifie.
- Etape identite validee uniquement si `city` et `uai` sont presents.
- Etapes classes, eleves et periodes calculees par count Supabase sous RLS.
- Validation inline si le nom de l'etablissement est vide.
- Erreur visible si l'update `establishments` ou `establishment_settings` echoue.
