## Decisions techniques prises

- Le mode demo reste preserve avec les anciens imports `@/data/demo`, isoles dans
  `StudentsDemo` et `StudentDetailDemo`.
- Le mode prod passe par `src/services/students.ts`, avec RLS naturelle et sans
  filtre manuel `establishment_id`.
- Les routes prod utilisent des view models Supabase (`StudentListItem`,
  `StudentDetail`) au lieu d'essayer d'adapter les composants demo couples aux
  types applicatifs historiques.
- La liste charge les eleves, classes, placements, entreprises et referents en
  lots fixes. Aucun N+1.
- La fiche detail charge l'eleve puis les donnees rattachees en parallele :
  classe, placement, visites, documents, alertes, entreprise, tuteur, referent.

## Architecture des donnees fetched

```text
/students
  -> isDemoMode()
     -> StudentsDemo -> data/demo
     -> StudentsSupabase -> fetchStudents()
        -> students / classes / placements / companies / teachers

/students/$id
  -> isDemoMode()
     -> StudentDetailDemo -> data/demo
     -> StudentDetailSupabase -> fetchStudentById(id)
        -> student
        -> class / placements / visits / documents / alerts
        -> company / tutor / referent
```
