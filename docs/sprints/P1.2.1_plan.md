# P1.2.1 - Plan d'attaque

## Objectif

Rendre le premier tenant réellement autonome après invitation :

- le DDFPT/admin peut corriger l'identité de son établissement depuis `/settings` ;
- le dashboard d'un tenant vide affiche une checklist de configuration actionnable au lieu de donner l'impression que l'app est vide.

## Lecture du brief

Brief lu : `docs/briefs/P1.2.1_tenant_identity_setup_checklist.md`.

Contraintes retenues :

- aucune migration SQL ;
- aucune policy RLS ajoutée ;
- mode démo intact ;
- maximum 4 requêtes de comptage pour la checklist ;
- save settings sans faux succès si une des deux écritures échoue ;
- checklist mobile-first et masquée automatiquement à 5/5.

## Surfaces concernées

### 1. `src/components/SetupChecklist.tsx`

Créer un composant pur.

Plan de rendu :

- props `steps` strictement conformes au brief ;
- `return null` si toutes les étapes sont faites ;
- `Card` + `Badge` + `Button` + `Link` TanStack ;
- icônes `CheckCircle2` / `Circle` ;
- layout mobile en pile verticale, desktop en liste compacte ;
- CTA par étape non terminée quand `cta` est fourni.

### 2. `src/services/dashboard.ts`

Étendre `DashboardData` avec :

```ts
setupChecklist: {
  identityComplete: boolean
  classesCount: number
  studentsCount: number
  periodsCount: number
}
```

Plan de données :

- récupérer l'établissement courant via le scope actif ;
- calculer `identityComplete` à partir de `city` et `uai` ;
- compter `classes`, `students`, `pfmp_periods` avec `select('*', { count: 'exact', head: true })` ;
- appliquer le scope établissement avec `applyEstablishmentScope` ;
- faire ces lectures même si aucune période active n'existe.

Nuance importante :

Le dashboard actuel affiche directement un EmptyState quand `currentPeriod` est `null`. Pour un tenant vide, c'est justement là que la checklist doit apparaître. Le service devra donc toujours retourner `setupChecklist`, et la route devra afficher la checklist avant l'empty state période.

### 3. `src/routes/dashboard.tsx`

Plan UI :

- importer `SetupChecklist` ;
- construire les 5 étapes depuis `data.setupChecklist` ;
- rendre la checklist en haut du contenu, y compris quand `currentPeriod` est absent ;
- conserver l'EmptyState "Aucune période PFMP" juste après la checklist ;
- ne pas modifier la structure existante du dashboard quand une période existe ;
- en mode démo, passer 5 étapes `done: true` pour que le composant retourne `null`.

Routes CTA prévues :

- identité : `/settings`
- classe : `/classes`
- élève : `/students`
- période : `/pfmp-periods`

### 4. `src/services/settings.ts`

Étendre `UpdateSettingsInput` avec :

- `name?: string`
- `city?: string | null`
- `uai?: string | null`

Plan mutation :

- préparer deux opérations :
  - `establishments.update(...)` si l'identité est dans le payload ;
  - `establishment_settings.upsert(...)` dans tous les cas ;
- exécuter avec `Promise.allSettled` ;
- si une opération échoue, lever une erreur claire qui mentionne quelle partie a échoué ;
- ne retourner un succès que si les deux opérations attendues ont réussi ;
- retourner `{ establishment, settings }` pour que la route mette son état local à jour ;
- conserver `logAuditAsync` après succès global uniquement.

### 5. `src/routes/settings.tsx`

Plan UI et état :

- ajouter `name`, `city`, `uai` en `useState` ;
- initialiser ces states depuis `fetchTenantSettings()` ;
- remplacer les `ReadonlyField` par des `Input` éditables ;
- supprimer `ReadonlyField` si plus utilisé ;
- validation soft : nom vide interdit ;
- inclure `name`, `city`, `uai` dans `UpdateSettingsInput` ;
- après save, mettre à jour `data.establishment` et `data.settings` avec le retour service ;
- afficher erreur inline dans la zone RGPD/save existante.

## Ordre d'implémentation prévu

1. Créer `SetupChecklist.tsx`.
2. Étendre `DashboardData` et `fetchDashboardData()`.
3. Brancher la checklist dans `/dashboard`, avec cas sans période active.
4. Étendre `UpdateSettingsInput` et `updateTenantSettings()`.
5. Brancher les inputs identité dans `/settings`.
6. Écrire `docs/sprints/P1.2.1_rapport.md`.
7. Lancer validations.

## Validations prévues

Commandes :

```bash
npm run typecheck
npm run build
git diff --check
```

Tests manuels si session prod disponible :

1. `/dashboard` tenant vide : checklist visible, compte activé OK, autres étapes à faire.
2. CTA identité -> `/settings`.
3. `/settings` : nom vide bloqué.
4. `/settings` : ville + UAI sauvegardés puis persistants après reload.
5. Retour `/dashboard` : étape identité passée en OK.
6. Mode démo : dashboard sans checklist visible.

## Risques identifiés

- Le dashboard actuel court-circuite sur l'absence de période active : je corrigerai le placement UI sans supprimer l'EmptyState existant.
- L'update en deux tables n'est pas transactionnel : je respecterai le brief avec `Promise.allSettled` et aucun faux message "saved".
- Si RLS bloque `establishments.update`, je ne contournerai pas côté client ; je documenterai l'erreur dans le rapport.

## Estimation

3 à 5 heures comme prévu dans le brief.
