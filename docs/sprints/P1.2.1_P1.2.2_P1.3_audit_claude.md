### Revue de code

**`SetupChecklist.tsx`** — composant pur, props bien typées (`SetupChecklistStep`), retourne `null` quand tout est `done` → mode démo invisible (Jean Moulin a 5/5 done). Aucun side-effect.

**`buildSetupSteps` et `buildDemoSetupSteps`** — séparés proprement. Les CTAs des 4 étapes pointent sur les bonnes routes : identity → `/settings`, class → `/classes`, student → `/students`, period → `/pfmp-periods`.

**`services/dashboard.ts`** — ajoute `setupChecklist` au type `DashboardData`. Counts via `count: 'exact', head: true`. RLS naturelle (pas de filtre `establishment_id` côté client).

**`services/settings.ts`** — `updateTenantSettings` exécute `Promise.allSettled` parallèle sur `establishments.update` et `establishment_settings.upsert`. La fonction `unwrapSupabaseSettled` distingue les deux pour produire un message d'erreur précis. Atomicité respectée : si l'une des deux échoue, on throw, le frontend ne marque pas "saved".

**`routes/settings.tsx`** — les 3 `ReadonlyField` sont remplacés par `<Input>` éditables (`name`, `city`, `uai`). Validation soft : `cleanName === '' → setError('Le nom ne peut pas etre vide.')`. UAI passé en uppercase + `maxLength={8}`.

### Pattern à surveiller (non-blocker)

`<Link to="/path"><Button>...</Button></Link>` produit `<a><button>...</button></a>`, structurellement invalide HTML. Ce pattern existe déjà dans `dashboard.tsx` et fonctionne en pratique avec TanStack Router. À refactorer un jour mais **pas un blocker pour ce merge**.

- Pas de migration SQL ✅
- RLS naturelle (pas de policy ajoutée) ✅
- Atomicité du save ✅
- ≤ 4 count queries ✅


### Revue de code

**`lib/tenantAccess.ts`** — helper pur :
- `getTenantAccessSlug()` : `subdomain || slug` (fallback OK)
- `buildTenantPlatformUrl()` : `https://{slug}.pfmp-pilot.fr`
- `buildTenantAccessUrl()` : préfère `custom_domain` si `domain_verified === true`
- `getTenantAccessInfo()` : retourne `kind`, `url`, `platformUrl`, `slug`, `label`, `description`

**`TenantAccessCard.tsx`** — 3 boutons, tous `type="button"` explicites :

1. **`handleCopy`** : `navigator.clipboard.writeText(url)` avec fallback `<textarea>` + `document.execCommand('copy')`. Feedback visuel `copied: true` pendant 2.2s. Si échec : `setCopyError`.
2. **`handleOpen`** : `window.open(url, '_blank', 'noopener,noreferrer')`. Standard.
3. **`handleDownload`** : `qrRef.current.toDataURL('image/png')` → `<a>.click()`. La ref est forwardée par `qrcode.react@4` (vérifié dans `node_modules/qrcode.react/lib/index.d.ts` : `React.RefAttributes<HTMLCanvasElement>`).

**Intégration dans les routes** — pas de `<form>` parent, pas de `disabled`, pas de `pointer-events: none`, pas d'overlay détectable.

### Investigation "boutons qui marchent pas"

Audit statique exhaustif :

| Bouton | Vérifications | Verdict statique |
|---|---|---|
| Copier le lien | clipboard + fallback execCommand | Devrait fonctionner |
| Ouvrir l'espace | DNS résout `lycee-test.pfmp-pilot.fr → 216.150.x.x` | Devrait fonctionner |
| Télécharger QR | ref forwardée vers canvas natif (vérifié source) | Devrait fonctionner |

**Hypothèses sur le ressenti BraveHeart** :

1. **Popup blocker au 1er clic** post-chargement. Comportement browser.
2. **Confusion produit** : "Ouvrir l'espace" ouvre `https://lycee-test.pfmp-pilot.fr` qui peut afficher une page Vercel sans détecter le tenant côté app.
3. **Test en localhost preview** : le QR encode l'URL prod.

### Aucune correction de code appliquée

Le code livré est techniquement correct, build et typecheck passent. Si BraveHeart confirme un bouton précis non fonctionnel après merge, on ouvre un micro-sprint correctif `P1.2.2.1`.
