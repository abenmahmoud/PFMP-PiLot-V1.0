# P1.9 — Placements PFMP + AI Matchmaking

## Objectif

Livrer le coeur operationnel PFMP : periodes par classe, affectation eleve -> entreprise/tuteur/referent, workflow de statut, et suggestions algorithmiques des meilleures entreprises pour chaque eleve.

## Constat technique initial

- `pfmp_periods` et `placements` existent deja dans `0001_init.sql`.
- Le schema actuel est plus ancien que le brief :
  - `pfmp_periods` n'a pas encore `class_id`, `type`, `notes`, `archived_at`.
  - `placements` n'a pas encore `notes`, `archived_at`.
  - `placements.referent_id` reference historiquement `teachers.id`, pas `profiles.id`; le code existant (`services/placements.ts`) lit deja le referent via `teachers`.
- Les statuts existants sont encore les statuts metier historiques :
  - periode : `preparation`, `in_progress`, `completed`, `archived`
  - stage : `no_stage`, `found`, `pending_convention`, `signed_convention`, `in_progress`, `completed`, `interrupted`
- P1.9 doit donc aligner sans casse : migration additive et mapping UI `draft -> no_stage`, `confirmed -> found`, `cancelled -> interrupted`.

- score :

  - famille professionnelle : 20

## Risques et decisions

- Decision : ne pas remplacer l'enum `stage_status` par les statuts du brief, car trop de pages utilisent deja `no_stage/found/pending_convention/...`. Le workflow P1.9 sera mappé sur ces statuts pour garder la compatibilite.
- Decision : conserver `placements.referent_id` comme `teachers.id` pour respecter le schema initial et les services existants. Les checks permissions remonteront au `teachers.profile_id`.
- Risque : les migrations prod peuvent deja diverger du `0001_init.sql`; la migration sera additive et `IF NOT EXISTS`.
