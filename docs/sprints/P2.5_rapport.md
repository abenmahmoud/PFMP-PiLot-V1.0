# P2.5 — Commercialisation durable + onboarding multi-etablissement

## Verdict

READY_FOR_AUDIT

Ce sprint transforme l entree publique de PFMP Pilot AI en surface commerciale exploitable, tout en conservant les portails operationnels sur une route dediee. Il ajoute aussi une base propre pour capter les demandes pilotes et un runbook d onboarding reproductible apres reset DB.

## Resultat fonctionnel

- `/` devient une landing commerciale claire pour vendre PFMP Pilot AI a un lycee, un reseau ou un rectorat.
- `/portails` conserve le choix Admin / Professeur / Eleve pour les utilisateurs existants.
- `/devis` permet de deposer une demande de demonstration ou de pilote.
- `/securite-rgpd` explique l isolation tenant, le portail eleve sans email, les secrets serveur et les preuves/audit logs.
- Une table `sales_leads` stocke les demandes commerciales via une server function service-role.
- Un runbook de mise en route pilote documente le reset, la creation tenant, l import SIECLE, les entreprises, les placements, les visites et signatures.

## Decisions techniques

1. La page racine `/` n est plus un choix de portail mais une landing commerciale. C est le bon comportement pour vendre : un prospect doit comprendre l offre avant de voir les acces internes.

2. Le choix operationnel des portails est conserve sur `/portails`. Aucun flux admin, professeur ou eleve n est supprime.

3. Les demandes commerciales passent par `submitSalesLead()` cote serveur avec `createAdminClient()`. La table n expose pas d insertion publique RLS ; le navigateur n ecrit pas directement dans Supabase.

4. Le formulaire `/devis` a un honeypot `website` pour absorber les bots simples sans polluer la base.

5. En mode demo, `/devis` affiche un succes local sans dependance Supabase. Cela garde les demonstrations robustes meme sans environnement complet.

6. Le runbook impose de ne jamais reutiliser un vieux tenant de test pour un vrai pilote.

## Securite et multi-tenant

- `sales_leads` est hors tenant metier et lisible/modifiable uniquement par superadmin via RLS.
- Les donnees prospects sont separees des etablissements clients.
- Les routes publiques n utilisent pas `AppLayout` admin.
- Les flux internes restent sous `/admin`, `/prof`, `/eleve`.
