## Resume

P2.1 livre une premiere version operationnelle du module terrain rural-friendly :

- planning visites pour prof/referent sur `/prof/visits`
- detail terrain `/prof/visits/$id` avec demarrage, geolocalisation, CR, dictee Web Speech, structuration IA, photos, competences, fin de visite
- tournee du jour `/prof/visits/tour`
- supervision admin `/admin/visits` + planification depuis un placement
- liste admin des visites signalees `/admin/visits/flagged`
- PWA installable avec manifest + service worker manuel
- queue hors-ligne IndexedDB pour actions terrain
- migration additive compatible avec l'ancien schema `visits`

## Decisions techniques

1. Migration additive au lieu d'une recreation de `visits`.
   L'ancien schema existe deja et les routes `/visits/new` / `/visits/$id` l'utilisent. La migration ajoute les colonnes terrain (`placement_id`, `referent_id`, `scheduled_at`, `done_at`, `photos`, etc.) et etend `visit_status`.

2. PWA manuelle sans dependance.
   Pas de `vite-plugin-pwa` ajoute : manifest + `public/sw.js` suffisent pour ce sprint et limitent le risque build.

3. Transcription vocale gratuite en priorite.
   Le flux principal utilise Web Speech API cote navigateur. Le serveur structure le texte avec Claude si `ANTHROPIC_API_KEY` existe, sinon fallback heuristique testable.

4. Offline queue native.
   IndexedDB est utilise sans Dexie/idb-keyval. Les actions restent en queue si aucun handler de sync n'est fourni.

5. Photos.
   Compression client max 1200 px et upload prepare vers bucket prive `visit-photos`. Hors-ligne, la photo est stockee en IndexedDB et referencee dans le CR comme photo en attente.

## Securite

- Service-role seulement dans server functions.
- Verification caller via `auth.getUser(accessToken)` dans `_lib.getCallerProfile`.
- Isolation tenant via `resolveEstablishmentId` / `assertSameTenant`.
- Mutations visite avec audit logs : plan, start, report update, complete, flag, cancel/no-show, evaluation.
- Le referent ne mute que ses visites ou ses eleves rattaches.
- Admin/DDFPT/superadmin gardent la supervision tenant.

## Limitations connues

- L'upload photo offline est prepare, mais la resynchronisation automatique des blobs stockes localement devra etre durcie en P2.1.1 avec un handler specifique par action.
- Les coordonnees GPS des entreprises ne sont pas encore dans `companies`; l'optimisation tournee fonctionne surtout sur l'ordre et les adresses Google Maps, avec distance a 0 tant que lat/lng manque.
- La structuration Claude n'a pas ete testee avec une vraie cle `ANTHROPIC_API_KEY` dans cet environnement.
- Le bucket `visit-photos` est cree par migration, mais un audit Supabase doit confirmer les policies Storage finales.
