# P2.1 - Plan d'attaque - Visites terrain + PWA hors-ligne + transcription vocale

## Objectif

Livrer un module terrain utilisable par les referents PFMP : planning de visites, demarrage sur mobile, compte-rendu, photos, transcription vocale Web Speech, queue hors-ligne IndexedDB, synchronisation au retour reseau et vues admin de supervision.

## Etat lu avant code

- La table `visits` existe deja dans `0001_init.sql` avec le modele historique `student_id`, `teacher_id`, `date`, `contact_type`, `status = draft/validated/archived`.
- Les routes historiques `/visits/new` et `/visits/$id` existent et restent utilisees depuis les fiches eleves.
- `/prof/visits` est aujourd'hui un stub.
- P1.9 a deja introduit `placements` et les vues `/admin/placements`, `/prof/placements`.
- `_lib.ts` fournit deja `createAdminClient`, `safeHandlerCall`, `toPlainJson`, `validateUuid`, `getCallerProfile`, `insertAuditLog`.

## Decisions techniques

1. Migration compatible, pas de recreation destructive de `visits`.
   - Ajouter les colonnes P2.1 avec `ALTER TABLE ... ADD COLUMN IF NOT EXISTS`.
   - Garder `status` historique pour compatibilite avec les anciennes routes.
   - Introduire `field_status` pour le cycle terrain `planned/in_progress/completed/cancelled/no_show`.
   - Ajouter `placement_id`, `referent_id`, `scheduled_at`, `done_at`, `duration_minutes`, `location_lat/lng`, `summary`, `full_report`, `voice_transcript`, satisfactions, flags, `photos`, `archived_at`.

2. Service functions P2.1 dediees.
   - Creer `src/server/visits.functions.ts` avec `safeHandlerCall` partout.
   - Les mutations ecrivent `audit_logs`.
   - Permissions serveur strictes : admin/ddfpt/superadmin complet tenant, referent sur ses visites, principal lecture sur ses classes.

3. PWA sans dependance lourde.
   - Pas d'ajout `vite-plugin-pwa` pour eviter un risque build/dependance inutile.
   - Ajouter `manifest.webmanifest`, `sw.js` manuel et un petit composant d'enregistrement service worker.
   - Mettre en cache assets + shell + routes terrain simples.

4. Offline-first pragmatique.
   - `src/lib/offlineQueue.ts` en IndexedDB natif, sans Dexie/idb-keyval.
   - Queue typable et extensible : `visit.update`, `visit.complete`, `visit.add_evaluation`, `visit.photo_upload`.
   - Hook `useOfflineSync()` pour badge et sync manuelle/auto.

5. Transcription vocale.
   - Web Speech API cote client pour dictee gratuite.
   - Server function de structuration texte (`structureVisitTranscript`) avec heuristique locale si `ANTHROPIC_API_KEY` absent.
   - Exporter aussi `transcribeVisitAudio` en fallback serveur, mais le flux principal reste texte -> structure.

6. Photos terrain.
   - Capture via input camera/mobile, compression canvas max 1200 px, stockage local possible.
   - Upload Storage prepare dans `photoCapture.ts`; bucket `visit-photos` cree par migration si Supabase Storage disponible.

7. UI.
   - `/prof/visits` : planning semaine, compteur offline, visites du jour, liens detail/tour.
   - `/prof/visits/$id` : demarrer visite, timer, geoloc, photo, vocal, CR, competences, terminer.
   - `/prof/visits/tour` : ordre optimise + lien Google Maps.
   - `/admin/visits` : supervision, filtres, taux realisation.
   - `/admin/visits/flagged` : visites signalees.

## Ordre d'implementation

1. Migration SQL compatible + types `database.types.ts`.
2. PWA minimale : manifest, service worker, root registration.
3. Librairies client : `offlineQueue`, `geolocation`, `photoCapture`, `voiceRecording`.
4. Server functions : `visits.functions.ts`, `voiceTranscription.functions.ts`.
5. Composants UI : offline, visits, voice, photos, tour.
6. Routes `/prof/visits*` et `/admin/visits*`.
7. Sidebar prof/admin + helpers permissions.
8. Rapport P2.1, typecheck, build.

## Risques et garde-fous

- Le schema historique `visits.status` n'accepte probablement pas `planned`; on n'y injecte donc pas ces valeurs.
- L'appel Claude/Anthropic ne doit jamais bloquer le MVP : la structuration heuristique permet de tester sans cle.
- Le mode offline ne doit pas promettre un upload photo si le bucket/policies ne sont pas appliques : les actions restent en queue avec message clair.
- Les anciennes routes `/visits/new` et `/visits/$id` doivent continuer a compiler.
