## Anomalies détectées

### Anomalie #1 — Récursion RLS sur `current_establishment_id()` (BLOCKER)

**Symptômes observés (3 sources d'évidence convergentes) :**

1. **REST sous JWT user** (handoff Codex) :
   ```
   GET /rest/v1/profiles?select=*&id=eq.<uid>
   → HTTP 500 / code 54001 / "stack depth limit exceeded"
   GET /rest/v1/establishments?select=id,name,active&limit=5
   → même erreur
   ```

2. **SQL direct sous role `authenticated`** (cette session) :
   ```
   SELECT count(*) FROM students;
   → ERROR 54001
   → CONTEXT: SQL function "current_establishment_id" during startup [×250+]
   ```

3. **UI prod, console navigateur** (cette session) :
   ```
   [ERROR] [auth] fetchProfile: stack depth limit exceeded
   (assets/index-DJ2wCUsr.js)
   ```

**Cause racine confirmée par lecture du catalogue Postgres :**

La fonction `public.current_establishment_id()` est définie en **`SECURITY INVOKER`** (les autres helpers `current_role_app`, `is_superadmin`, `is_establishment_admin`, `is_referent_of`, `can_read_all_students` sont, eux, en `SECURITY DEFINER`). Son corps fait :

```sql
select coalesce(
  case when (select role = 'superadmin' from profiles where id = auth.uid()) then ...
       else null end,
  (select establishment_id from profiles where id = auth.uid())
);
```

Elle est invoquée depuis les policies suivantes (au moins) :

| Table | Policy | cmd | Expression contenant `current_establishment_id()` |
|---|---|---|---|
| `profiles` | `profiles_select` | SELECT | `is_superadmin() OR (id = auth.uid()) OR (establishment_id = current_establishment_id())` |
| `profiles` | `profiles_update` | UPDATE | `... OR (is_establishment_admin() AND (establishment_id = current_establishment_id()))` |
| `establishments` | `establishments_select` | SELECT | `is_superadmin() OR (id = current_establishment_id())` |
| `students` | `students_select` | SELECT | `is_superadmin() OR ((establishment_id = current_establishment_id()) AND (...))` |
| `students` | `students_update` | UPDATE | `is_superadmin() OR (is_establishment_admin() AND (establishment_id = current_establishment_id()))` |

Probablement aussi les policies analogues sur `classes`, `teachers`, `tutors`, `placements`, `companies`, etc. — à confirmer dans le sprint fix.

**Mécanique de la boucle (sous rôle `authenticated`) :**

1. Le client REST appelle `GET /profiles` avec JWT user.
2. Postgres évalue la policy `profiles_select`.
3. Le 3ᵉ disjoint `(establishment_id = current_establishment_id())` est évalué (Postgres ne peut pas court-circuiter de façon fiable une expression qui contient une fonction `STABLE` non-folding).
4. `current_establishment_id()` (INVOKER) fait `SELECT … FROM profiles WHERE id = auth.uid()`.
5. Cette sous-requête est elle-même soumise à RLS → **re-déclenche `profiles_select`** → retour étape 3.
6. Stack overflow → erreur Postgres `54001`.

Même cascade sur `establishments_select`, `students_select`, et toutes les policies qui combinent `current_establishment_id()` avec une lecture sur la table couverte par cette fonction (donc `profiles` direct, ou tout chemin transitif y menant).

**Pourquoi `is_superadmin()` ne plante pas, elle :** parce qu'elle est correctement définie en `SECURITY DEFINER` avec `search_path = public`, via la fonction `current_role_app()` également DEFINER. DEFINER bypasse RLS sur `profiles`. **Le bug est strictement localisé à `current_establishment_id()`.**

**Impact produit (vérifié) :**

- Tout JWT `authenticated` non-service-role qui tente de lire `profiles`, `establishments`, `students` (et probablement d'autres) → 500.
- L'UI prod fait `AuthProvider → fetchProfile()` post-login → erreur silencieuse → **fallback démo** (constaté : "Sophie Bernard / Lycée Jean Moulin", données mockées, sidebar "DÉMO : CHANGER DE RÔLE").
- **Aucun parcours métier réel n'est testable en prod, y compris superadmin.** L'app est en réalité en mode démo permanent jusqu'au fix.
- **Le superadmin switcher est absent** (constaté).

**Recommandation de fix (format prescriptif pour sprint suivant) :**

- **Sprint cible :** `P0.2.1-fix-rls-recursion`
- **Owner :** Codex (implémentation), Claude (audit), BraveHeart (merge)
- **Fichier source du bug :** `supabase/migrations/0004_rls_hardening.sql` (ou la migration qui définit `current_establishment_id`, à confirmer via `git log -- supabase/migrations`)
- **Fonction responsable :** `public.current_establishment_id()`
- **Hypothèse technique :** `SECURITY INVOKER` + lecture de `profiles` + utilisation depuis policies de `profiles` ⇒ récursion infinie.
- **Pattern de fix recommandé :** passer en `SECURITY DEFINER` avec `search_path` explicite, en miroir exact de `current_role_app()` et `is_superadmin()`. Migration ~15 lignes :

  ```sql
  -- supabase/migrations/<YYYYMMDDHHMMSS>_fix_current_establishment_id_recursion.sql
  CREATE OR REPLACE FUNCTION public.current_establishment_id()
  RETURNS uuid
  LANGUAGE sql
  STABLE
  SECURITY DEFINER
  SET search_path TO 'public'
  AS $function$
    select coalesce(
      case
        when current_role_app() = 'superadmin' then
          nullif(
            (auth.jwt() -> 'user_metadata' ->> 'active_establishment_id'),
            ''
          )::uuid
        else null
      end,
      (select establishment_id from profiles where id = auth.uid())
    );
  $function$;
  ```

  Note de simplification : le sous-`select role = 'superadmin' from profiles` original peut être remplacé par `current_role_app() = 'superadmin'` (réutilise une fonction DEFINER déjà sûre, économise un round-trip SQL, et clarifie la dépendance).

- **Effort estimé :** XS (≤30 minutes incluant tests)
- **Risque régression :** faible. Sémantique identique, seul le mode d'exécution change. Le `establishment_id` lu dans le fallback reste celui du user authentifié — pas d'élévation de privilège.
- **Définition de done du sprint fix :**
  1. Migration appliquée
  2. `GET /rest/v1/profiles?select=*&id=eq.<uid>` retourne 200 avec JWT user
  3. `GET /rest/v1/establishments?select=id,name,active&limit=5` retourne 200 avec JWT user
  4. `rls_isolation.sql` adapté : Tests 1–5 verts
  5. Login UI prod : disparition du fallback démo, données réelles affichées, switcher superadmin présent
  6. Console : plus d'erreur `fetchProfile: stack depth limit exceeded`
