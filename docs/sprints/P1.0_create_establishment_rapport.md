# P1.0 - Creation minimale d'etablissement

Date : 2026-05-11
Branche : `codex/p1-0-create-establishment`

## Objectif

Corriger le CTA superadmin "Nouvel etablissement" qui ne faisait rien, en
branchant un flux minimal de creation de tenant avant l'onboarding complet.

## Contexte

Dans `src/routes/superadmin.establishments.tsx`, les boutons :

- `Nouvel etablissement`
- `Creer un etablissement`

etaient de simples composants `<Button>` sans `Link`, sans `onClick`, et sans
route cible. L'utilisateur pouvait donc cliquer sans aucun effet.

## Changements

- Ajout de `src/routes/superadmin.establishments.new.tsx`
  - route `/superadmin/establishments/new` ;
  - garde superadmin ;
  - formulaire tenant : nom, ville, UAI, slug, couleur, annee scolaire, seuil de charge ;
  - creation refusee en mode demo ;
  - retour a `/superadmin/establishments` apres creation.

- Extension de `src/services/superadmin.ts`
  - `CreateEstablishmentInput` ;
  - `CreateEstablishmentResult` ;
  - `createEstablishment()` ;
  - insertion dans `establishments` avec `status='trial'` et `active=true` ;
  - insertion dans `establishment_settings` ;
  - audit log `establishment.created`.

- Mise a jour de `src/routes/superadmin.establishments.tsx`
  - CTA header vers `/superadmin/establishments/new` ;
  - CTA empty state vers `/superadmin/establishments/new`.

## Hors scope volontaire

- Pas d'invitation DDFPT/admin.
- Pas de magic link.
- Pas de route detail `/superadmin/establishments/$id`.
- Pas de creation d'utilisateurs.
- Pas de workflow onboarding complet.

Ces elements restent le sprint P1 onboarding complet.

## Validation

- `npm run build` : OK.
- `npm run typecheck` : OK apres generation du route tree par le build.
- `git diff --check` : a executer avant commit final.

## Risques / notes

- Si la creation echoue en prod, la cause probable sera une policy RLS
  `establishments` / `establishment_settings` qui refuse l'insert. Dans ce cas
  l'erreur Supabase remontera dans le formulaire.
- Le slug est genere cote client pour le MVP. La contrainte unique en base reste
  la protection finale.

## Prochaine etape

Apres merge, tester en prod :

1. se connecter superadmin ;
2. aller sur `/superadmin/establishments` ;
3. cliquer `Nouvel etablissement` ;
4. creer un tenant test ;
5. verifier qu'il apparait dans la liste ;
6. supprimer/archiver le tenant test manuellement si besoin.

Ensuite : reprendre P0.10 hardening tenant actif avant P1 onboarding complet.
