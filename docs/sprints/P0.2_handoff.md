### Anomalie bloquante detectee via REST authentifie

Apres login reussi, lecture du profil :

```text
GET /rest/v1/profiles?select=*&id=eq.a6fff903-f90a-42ea-8f00-f93d988d6046
Authorization: Bearer <access-token>
```

Resultat :

```json
{
  "code": "54001",
  "details": null,
  "hint": "Increase the configuration parameter \"max_stack_depth\" (currently 2048kB), after ensuring the platform's stack depth limit is adequate.",
  "message": "stack depth limit exceeded"
}
```

Status HTTP : `500`.

Lecture `establishments` avec le meme JWT :

```text
GET /rest/v1/establishments?select=id,name,active&limit=5
```

Resultat identique :

```json
{"code":"54001","message":"stack depth limit exceeded"}
```

Analyse Codex : probable recursion RLS entre `is_superadmin()`,
`current_establishment_id()` et les policies qui relisent `profiles`.
Impact probable : apres login UI, `AuthProvider -> buildAuthState() ->
fetchProfile()` risque de produire `profile = null` avec erreur, donc le
superadmin switcher peut ne pas apparaitre meme si Auth a reussi.

### Audit RLS via vue `rls_audit`

Avec le JWT du superadmin de test :

```text
GET /rest/v1/rls_audit?select=table_name,rls_enabled,policy_count
```

Resultat : `200`, toutes les lignes retournees ont `rls_enabled: true`
et `policy_count > 0`.

Filtre equivalent a `unprotected_tables = 0` :

```text
GET /rest/v1/rls_audit?select=table_name&or=(rls_enabled.eq.false,policy_count.eq.0)
```

Resultat :

```json
[]
```

Conclusion partielle : `unprotected_tables = 0` confirme via REST.
