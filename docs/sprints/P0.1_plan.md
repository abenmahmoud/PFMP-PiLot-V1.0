# P0.1 — Plan d’attaque

## Objectif du sprint

Préparer la migration progressive de la démo vers Supabase sans casser la vitrine commerciale actuelle.

Ce sprint ne migre pas les pages métier vers Supabase. Il fige la démo comme référence produit, vérifie le comportement de `useCurrentUser`, confirme l’existence d’un `EmptyState` réutilisable et produit un audit clair des fichiers qui importent encore `@/data/demo`.

## Documents lus avant plan

- `docs/briefs/P0.1_copy_demo_reference.md`
- `AGENTS.md`, notamment la section Hub projet et les conventions de collaboration
- `docs/VISION.md`, source de vérité produit finale

## Contraintes non négociables respectées

- Ne pas modifier `src/data/demo.ts`.
- Ne pas migrer les 23 fichiers métier qui importent `@/data/demo`.
- Ne pas contourner la RLS.
- Ne pas utiliser de `service_role` côté client.
- Préserver le mode démo `VITE_DEMO_MODE=true`.
- Garder la démo commerciale intacte.
- Produire un rapport d’exécution `docs/sprints/P0.1_rapport.md`.
- Terminer avec `npm run typecheck` et `npm run build` verts, sauf blocker documenté.

## Plan technique proposé

### 1. Vérifier l’état réel du repo

Actions :

- Lire `src/data/demo.ts` pour confirmer la source démo actuelle.
- Lire `src/lib/useCurrentUser.ts` pour vérifier l’état du bridge démo/prod.
- Lire `src/components/EmptyState.tsx` pour confirmer son API.
- Rechercher tous les imports `@/data/demo`.
- Vérifier l’existence des scripts `typecheck`, `build`, `check`.

Risque : faible. Cette étape est purement exploratoire.

### 2. Créer la référence figée de la démo

Fichier prévu :

- `data-reference/demo.reference.ts`

Méthode :

- Copier le contenu de `src/data/demo.ts` sans le modifier.
- Ajouter uniquement l’en-tête demandé dans le brief.
- Ne jamais importer ce fichier dans le code applicatif.

Risque : faible. La copie n’impacte pas le runtime.

### 3. Auditer et ajuster `useCurrentUser`

Fichier à inspecter :

- `src/lib/useCurrentUser.ts`

Comportements attendus :

- Mode démo : retour de l’utilisateur démo sélectionné via le switcher de rôle.
- Mode prod avec Supabase configuré et session active : retour du profil Supabase au shape attendu par les composants.
- Mode prod sans session : retour d’un état utilisable par `RoleGuard` / redirection `/login`, sans crash.

Approche :

- Si le bridge est déjà correct, ne pas refactorer inutilement.
- Si un trou est détecté, patch minimal ciblé.
- Ne pas migrer les pages métier.
- Ne pas casser le switcher de rôle en démo.

Risque principal : casser le shape `Profile` attendu par `Sidebar`, `RoleGuard`, `Topbar` ou `index.tsx`. Mesure : garder une compatibilité stricte avec les types existants.

### 4. Vérifier `EmptyState`

Fichier :

- `src/components/EmptyState.tsx`

Actions :

- Confirmer qu’il existe.
- Vérifier qu’il est mobile-first et sobre.
- Vérifier si son API est compatible avec l’API souhaitée dans le brief.
- Si l’API diffère légèrement mais que les usages existants fonctionnent, éviter une refonte large.
- Si nécessaire, ajouter une compatibilité `action` minimale sans casser les usages actuels.

Risque : faible à moyen si l’API est utilisée sur plusieurs pages. Patch minimal uniquement.

### 5. Auditer les routes/fichiers qui importent `@/data/demo`

Objectif : produire une liste utile pour P0.3 à P0.9, pas migrer.

Méthode :

- Rechercher tous les imports directs de `@/data/demo`.
- Pour chaque fichier : identifier la route ou le composant concerné, les entités lues, le risque en prod vide, la criticité et un sprint cible conseillé.
- Noter clairement les pages qui afficheraient de fausses données Jean Moulin en prod.
- Noter les pages susceptibles de crasher avec 0 rows, notamment celles utilisant `find(...)!` ou des hypothèses fortes.

Livrable : section dédiée dans `docs/sprints/P0.1_rapport.md`.

### 6. Créer le rapport d’exécution

Fichier prévu :

- `docs/sprints/P0.1_rapport.md`

Contenu conforme au brief :

- Fichiers modifiés/créés.
- Décisions techniques prises.
- Comportement de `useCurrentUser` après sprint.
- Liste exhaustive des fichiers à migrer.
- Ce qui reste mocké ou TODO.
- Blockers éventuels.
- Preuves : typecheck, build, mode démo, mode prod.

### 7. Vérifications finales

Commandes prévues :

```bash
npm run typecheck
npm run build
```

Si possible selon l’environnement :

```bash
VITE_DEMO_MODE=true npm run dev
```

Pour le mode prod, si les credentials et variables Supabase sont disponibles localement, tester le login avec :

- `test@pfmp-pilot.fr`

Sinon documenter honnêtement que le test prod réel n’a pas pu être exécuté et préciser ce qui a été vérifié statiquement.

## Fichiers qui pourront être modifiés après validation du plan

Prévision maximale :

- `data-reference/demo.reference.ts` — création de la référence figée.
- `src/lib/useCurrentUser.ts` — patch minimal si nécessaire.
- `src/components/EmptyState.tsx` — patch minimal si nécessaire.
- `AGENTS.md` — uniquement si `EmptyState` est créé ou si son API est documentée.
- `docs/sprints/P0.1_rapport.md` — rapport final.

Aucun autre fichier applicatif ne sera touché sans validation.

## Points de vigilance

- Ne pas confondre référence figée et runtime démo.
- Ne pas remplacer `@/data/demo` dans les pages maintenant.
- Ne pas supprimer le switcher de rôle tant que le mode démo est nécessaire.
- Ne pas introduire de dépendance.
- Ne pas faire d’appel Supabase généralisé dans les pages métier.
- Ne pas masquer le problème actuel : les pages affichent encore des données démo en prod tant qu’elles ne sont pas migrées.

## Critères de réussite

- Plan committé avant tout code applicatif.
- Référence démo créée.
- `useCurrentUser` validé ou corrigé minimalement.
- `EmptyState` validé.
- Audit des imports `@/data/demo` livré.
- Rapport final créé.
- Typecheck et build verts, ou blocker explicite documenté.
