# P0.4 - Plan d'execution

## Objectif

Sortir les pages eleves de la demo en production. En mode demo, conserver le
comportement actuel. En mode Supabase, `/students` et `/students/$id` lisent
les donnees reelles filtrees par RLS.

## Perimetre

- `src/services/students.ts` : nouveau service de lecture Supabase.
- `src/routes/students.tsx` : liste eleves Supabase + recherche/filtres.
- `src/routes/students.$id.tsx` : detail eleve Supabase + 404 propre.
- `docs/sprints/P0.4_rapport.md` : rapport final.

## Architecture retenue

```text
route students
  -> isDemoMode()
     -> rendu demo existant
     -> rendu Supabase
        -> useAuth()
        -> fetchStudents() / fetchStudentById()
        -> getSupabase()
        -> RLS naturelle
```

## Requetes prevues

- Liste :
  - `students.select(*)`
  - `classes.select(id,name,level,specialty)`
  - `placements.select(student_id,status,company_id,referent_id)`
  - `companies.select(id,name)`
  - `teachers.select(id,first_name,last_name)`
- Detail :
  - `students.maybeSingle()`
  - `classes.maybeSingle()`
  - `placements` actifs/recents
  - `companies` / `tutors` si placement
  - `visits`, `documents`, `alerts` lies a l'eleve

Aucun filtre manuel `establishment_id`. La RLS reste la frontiere de securite.

## Empty states

- Session absente : message "Session requise".
- Liste vide : "Aucun eleve dans cet etablissement".
- Detail introuvable : "Eleve introuvable ou inaccessible".
- Sections detail vides : aucun stage, aucune visite, aucun document, aucune alerte.

## Validation

- `npm run typecheck`
- `npm run build`
- Smoke manuel apres preview/prod : ouvrir `/students`, filtrer, ouvrir une fiche.

## Limites assumees P0.4

- Mutations/import eleves hors scope.
- Pages classes/companies/documents encore demo jusqu'aux sprints suivants.
- Pas d'anticipation referentiel P2.
