# P0.2.1 - Plan d'execution

## Comprehension du bug

`current_establishment_id()` est actuellement executee en `SECURITY INVOKER`
et lit `profiles`. Quand elle est appelee depuis les policies de `profiles`
ou de tables qui dependent d'elle, la lecture de `profiles` re-declenche les
memes policies et provoque `stack depth limit exceeded`.

## Approche retenue

Appliquer exactement le pattern prescrit dans `docs/sprints/P0.2_rapport.md` :
recreer `public.current_establishment_id()` en `SECURITY DEFINER` avec
`SET search_path TO 'public'`, en reutilisant `current_role_app()` pour le cas
superadmin. La semantique reste identique : claim
`active_establishment_id` prioritaire pour superadmin, fallback sur
`profiles.establishment_id` pour l'utilisateur authentifie.

## Fichier a creer

- `supabase/migrations/<YYYYMMDDHHMMSS>_fix_current_establishment_id_recursion.sql`

Le timestamp sera genere au moment de l'implementation, au format Supabase
`YYYYMMDDHHMMSS`.

## Plan d'application

- En local : creer uniquement la migration dans le repo, sans modifier les
  migrations historiques.
- Application DB : je n'ai pas de secret `DATABASE_URL`, pas de `psql`, pas de
  Supabase MCP expose dans cet environnement. Je prevois donc un
  `HANDOFF Claude` pour appliquer la migration en prod via Supabase MCP
  `apply_migration`, ou pour qu'Adel l'applique via Supabase dashboard/CLI.
- Apres application : je peux verifier les criteres REST avec la publishable key
  et le compte `test@pfmp-pilot.fr`; les validations SQL/UI qui exigent MCP
  seront deleguees si les outils restent indisponibles.
