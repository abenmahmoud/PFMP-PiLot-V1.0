## Ce qui est livré

### Annuaire professeurs

- `/teachers` affiche maintenant les professeurs avec stats : classes suivies comme professeur principal et nombre d'élèves référencés.
- Le bouton `Ajouter` ouvre une modale de création/édition.
- Email optionnel : un professeur peut être créé sans compte SaaS.
- Si un email est fourni avec invitation, le flux réutilise l'invitation existante et lie automatiquement la ligne `teachers` au `profile_id`.
- Import CSV/XLSX en modale avec preview 5 lignes, parsing client-side et appels serveur par lot.
- Archivage soft-delete avec nettoyage automatique des affectations `classes.principal_id` et `students.referent_id`.
- `/teachers/$id` affiche la fiche détaillée, les classes où le professeur est principal, les élèves dont il est référent et l'historique audit.

### Affectations

- `assignReferentToStudent` est implémentée : admin/DDFPT/superadmin ou professeur principal de la classe peuvent affecter un référent.
- `assignClassPrincipal` est implémentée : admin/DDFPT/superadmin peuvent affecter ou retirer le professeur principal d'une classe.
- `assignPlacementToStudent` reste volontairement stub P1.8, mais vérifie maintenant la session avant de répondre "non implémenté".

### UI métier

- `/classes/$id` contient une section `Professeur principal` au-dessus des codes élèves.
- `/students/$id` contient une card `Référent PFMP` avec affectation via `SelectTeacher`.
- `SelectTeacher` est réutilisable, avec recherche et filtre par rôle.

## Sécurité

- Toutes les nouvelles server functions utilisent `safeHandlerCall`.
- Les mutations passent par `createAdminClient`, puis vérifient `auth.getUser(accessToken)` via `getCallerProfile`.
- Les droits tenant sont vérifiés côté serveur avant chaque mutation.
- Audit log écrit pour :
  - `teacher.created`
  - `teacher.updated`
  - `teacher.archived`
  - `teacher.bulk_imported`
  - `student.referent_assigned`
  - `class.principal_assigned`
- Aucun fichier uploadé professeur n'est stocké : parsing en mémoire navigateur uniquement.

## Décisions techniques

- Pas de migration SQL : la colonne `teachers.discipline`, `classes.principal_id` et `students.referent_id` sont considérées présentes en prod selon le brief.
- Pas d'upsert SQL `establishment_id,email` car le schéma local ne garantit pas d'index unique. L'import cherche donc d'abord un professeur actif par email dans le tenant, puis update ou insert.
- `inviteUserToEstablishmentUnsafe` est exportée pour éviter de dupliquer la logique d'invitation dans `teachers.functions.ts`.
- Le rattachement automatique `teachers.profile_id` par email a été renforcé dans `invitations.functions.ts`.

## Hors périmètre assumé

- Affectation de placement/stage : conservée pour P1.8.
- Gestion fine des professeurs cross-tenant par superadmin global : les écrans `/teachers` restent centrés tenant.
- Import supérieur à 200 lignes : rejeté côté server function pour garder un payload maîtrisé.
