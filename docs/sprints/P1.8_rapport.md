## Multi-tenant

Les mutations ne font jamais confiance à un `establishment_id` client :

- admin/DDFPT : l'établissement vient du profil appelant ;
- superadmin : l'établissement actif est obligatoire ;
- toute ressource ciblée est vérifiée contre le tenant résolu.

Les inserts injectent `establishment_id` côté serveur.

## Import entreprises

Formats :

- CSV séparateur `;` ou `,`
- XLS/XLSX via dynamic import `xlsx`

Colonnes reconnues :

- Nom / Entreprise / Raison sociale
- SIRET
- Adresse
- Code postal
- Ville
- Téléphone
- Email
- Site web
- Secteur
- Famille métier

Déduplication :

- priorité au SIRET si présent ;
- sinon `name + city`.

Sécurité :

- fichier parsé en mémoire navigateur ;
- fichier non stocké ;
- max 200 lignes par appel serveur ;
- dry-run obligatoire dans l'UI avant le commit.

## Risques / points à auditer

- `findExistingCompanyForImport()` utilise `ilike(name)` + `ilike(city)` en fallback si SIRET absent. C'est acceptable pour MVP, mais un matching plus strict/normalisé pourra être ajouté si les imports réels sont sales.
- Les mutations n'ont pas encore de tests Playwright authentifiés ; audit Claude devra valider sur preview avec compte admin.
- L'archivage tuteur masque le tuteur de la fiche active. La restauration tuteur existe côté server function mais n'est pas exposée dans l'UI, car le brief demandait seulement archiver.
