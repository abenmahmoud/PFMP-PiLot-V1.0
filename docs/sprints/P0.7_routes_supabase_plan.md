# P0.7 — Routes listes Supabase

## Objectif

Migrer les routes listes `classes`, `teachers`, `pfmp-periods` et `documents` en lecture Supabase, sans casser le mode demo.

## Contexte

Les PR precedentes ont etabli le pattern dual-mode :

- `isDemoMode()` conserve l'ancien rendu avec `@/data/demo`.
- Le mode Supabase utilise `useAuth()`, un service dedie dans `src/services/*`, et des etats `loading/error/empty`.
- La securite tenant/referent reste portee par la RLS Supabase, pas par des filtres client.

## Plan d'attaque

1. Ajouter `src/services/classes.ts` pour composer classes, principal, effectifs et eleves sans stage.
2. Ajouter `src/services/teachers.ts` pour composer enseignants, classes principales et charge d'affectation.
3. Ajouter `src/services/pfmpPeriods.ts` pour composer periodes, classes rattachees, taux d'affectation, taux de visite et documents manquants.
4. Ajouter `src/services/documents.ts` pour composer documents, eleve, periode et entreprise.
5. Refondre chaque route en `Demo` + `Supabase` en conservant l'UI existante autant que possible.
6. Ajouter des empty states qualitatifs pour les tenants vides.
7. Valider `npm run typecheck` et `npm run build`.

## Contraintes non negociables

- Mode demo intact.
- Aucune mutation Supabase dans ce sprint.
- Aucun contournement RLS avec `establishment_id` injecte depuis le client.
- Boutons de creation/upload/export desactives en mode Supabase tant que les mutations ne sont pas implementees.

## Resultat attendu

Les quatre routes n'affichent plus les fixtures demo en mode Supabase. Elles affichent soit les donnees reelles du tenant, soit un empty state propre si la base est vide.
