# P1.5 - Rapport Codex - Portail eleve sans email

Date : 12 mai 2026
Branche : `feature/p1-5-student-portal-no-email`
Verdict Codex : `READY_FOR_AUDIT`

## Objectif livre

Un eleve peut ouvrir `/eleve`, saisir son code PFMP ou scanner le QR code `/eleve?code=PFMP-XXXX-XXXX`, puis acceder a `/eleve/dashboard` sans compte email, sans Supabase Auth et sans JWT.

La session eleve est stockee uniquement en `sessionStorage`, disparait a la fermeture de l'onglet, et permet d'afficher une fiche de stage en lecture seule.

## Server functions

### `validateStudentCode`

Entree :

```ts
{ code: string }
```

Sortie :

```ts
{ ok: true, session: StudentPublicSession }
// ou
{ ok: false, error: 'CODE_INVALID' | 'CODE_REVOKED' | 'CODE_EXPIRED' }
```

Logique :

- normalise le code avec la meme logique que P1.4
- hash SHA-256 avec `STUDENT_ACCESS_CODE_PEPPER`, fallback `SUPABASE_SERVICE_ROLE_KEY`
- cherche le code dans `student_access_codes`
- refuse les codes absents, revoques ou expires
- charge l'eleve, sa classe et son etablissement
- met a jour `last_used_at`
- ecrit `audit_logs.action = "student_portal.code_validated"` avec `user_id = null`
- retourne une session publique sanitisee pour Seroval

### `getStudentPublicDashboard`

Entree :

```ts
{ studentId: string }
```

Sortie :

```ts
StudentPublicDashboard | null
```

Logique :

- verifie qu'un code actif existe encore pour l'eleve
- retourne `null` si aucun code actif, eleve archive ou session invalide
- charge l'eleve, la classe, le placement le plus recent, l'entreprise et le tuteur
- retourne uniquement des donnees de lecture publique
- aucun audit log pour eviter le spam a chaque refresh

Tous les retours passent par `sanitizeForSeroval()`.

## Routes publiques

### `/eleve`

- pas de `AppLayout`
- page publique centree, branding PFMP Pilot AI
- pre-remplit le champ depuis `?code=`
- auto-submit si un code est present dans l'URL
- normalisation visuelle du format `PFMP-XXXX-XXXX`
- erreurs FR :
  - code invalide
  - code revoque
  - code expire
- succes : ecriture `sessionStorage.pfmp_student_session` puis navigation `/eleve/dashboard`

`eleve.tsx` gere aussi le cas route parente via `useMatchRoute()` + `<Outlet />`, pour laisser `/eleve/dashboard` se rendre correctement.

### `/eleve/dashboard`

- pas de `AppLayout`
- lit `sessionStorage`
- redirect `/eleve` si session absente ou malformee
- charge le dashboard public via server function
- affiche :
  - card "Mon stage"
  - classe, formation, statut
  - periode si disponible
  - entreprise et tuteur si disponibles
  - card "Mes informations"
- si aucun placement : message "Votre stage n'est pas encore assigne..."
- bouton "Se deconnecter" : supprime la session et retourne a `/eleve`

## Securite

- Zero `auth.signIn()` Supabase pour l'eleve.
- Zero compte `auth.users` eleve.
- Zero JWT Supabase cote eleve.
- Zero cookie.
- `sessionStorage` uniquement.
- Le code complet n'est jamais stocke dans la session.
- Les server functions utilisent la service-role uniquement cote serveur.
- Le dashboard ne sert les donnees que si l'eleve possede encore un code actif.
- Dashboard lecture seule absolue.
- Aucun hash ni hint de code n'est retourne au portail eleve.

## Limites assumees

- La session eleve est volontairement legere : elle ne contient pas de signature serveur ni JWT.
- `getStudentPublicDashboard` accepte `studentId` mais ne retourne les donnees que si un code actif existe encore.
- Les placements/companies/tutors sont geres comme facultatifs car la base pilote peut etre vide.
