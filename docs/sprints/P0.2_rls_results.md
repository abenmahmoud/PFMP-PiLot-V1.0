## 5. 🎯 BONUS — Analyse du bug RLS critique (par lecture statique)

### Cause racine identifiée

Codex a soupçonné une **récursion RLS entre `is_superadmin()`,
`current_establishment_id()` et les policies sur `profiles`/`establishments`**.
Cette hypothèse est **CONFIRMÉE** par lecture des migrations 0001-0003.

### Code coupable

Migration `0003_auth_jwt.sql`, lignes ~30-50 :

```sql
create or replace function current_establishment_id()
returns uuid
language sql stable    -- ❌ MANQUE security definer
as $$
  select coalesce(
    case
      when (select role = 'superadmin' from profiles where id = auth.uid()) then
        -- ⚠️ Cette lecture de profiles déclenche la policy profiles_select
        nullif(
          (auth.jwt() -> 'user_metadata' ->> 'active_establishment_id'),
          ''
        )::uuid
      else null
    end,
    (select establishment_id from profiles where id = auth.uid())
    -- ⚠️ Idem ici : 2e lecture de profiles
  );
$$;
```

À comparer avec `is_superadmin()` (migration 0001), correctement définie :

```sql
create or replace function is_superadmin()
returns boolean
language sql stable security definer    -- ✅ PRESENT
set search_path = public
as $$
  select coalesce(current_role_app() = 'superadmin', false);
$$;
```

### Pourquoi ça boucle

La policy `establishments_select` (migration 0001) :

```sql
create policy establishments_select on establishments for select
  using (is_superadmin() or id = current_establishment_id());
```

Et la policy `profiles_select` (migration 0001) contient probablement aussi
un `is_superadmin()` ou une logique similaire qui déclenche une lecture de
`profiles`.

Le cycle :

1. `SELECT * FROM establishments`
2. Policy `establishments_select` invoque `current_establishment_id()`
3. `current_establishment_id()` (sans `security definer`) fait
   `SELECT role FROM profiles WHERE id = auth.uid()`
4. Postgres applique `profiles_select` sur cette lecture
5. Si `profiles_select` invoque elle-même `current_establishment_id()`
   ou utilise `(select ... from profiles)` → récursion
6. Stack atteint 2048kB → `54001 stack depth limit exceeded`

### Patch proposé (à valider en P0.3 ou sprint dédié)

**Migration `supabase/migrations/0005_fix_rls_recursion.sql`** (à créer) :

```sql
-- =====================================================================
-- Fix bug RLS recursion : current_establishment_id() doit etre
-- security definer pour court-circuiter les policies sur profiles
-- =====================================================================

create or replace function current_establishment_id()
returns uuid
language sql stable
security definer       -- ⭐ AJOUT CRITIQUE
set search_path = public
as $$
  select coalesce(
    case
      when (select role = 'superadmin' from profiles where id = auth.uid()) then
        nullif(
          (auth.jwt() -> 'user_metadata' ->> 'active_establishment_id'),
          ''
        )::uuid
      else null
    end,
    (select establishment_id from profiles where id = auth.uid())
  );
$$;

-- Verif : la fonction doit avoir prosecdef = true
-- select proname, prosecdef from pg_proc where proname = 'current_establishment_id';
```

### Impact estimé

- ⚠️ **Sans ce fix, P0.3 (dashboard lit Supabase) est BLOQUÉ** : le hook
  `useCurrentUser` qui essaye de fetch le profil va crasher en prod.
- ⚠️ **Tous les sprints P0+ sont impactés** tant que ce bug n'est pas patché.
- ✅ Le patch est **simple et chirurgical** : ajouter 2 mots
  (`security definer`) dans la définition de `current_establishment_id()`.
- ✅ Aucun risque de régression sur Sprint 1/2/3 puisque c'est un fix
  comportemental, pas un changement d'interface.
