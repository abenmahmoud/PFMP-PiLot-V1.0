## Decisions techniques prises

- Le dashboard garde deux chemins explicites : `DashboardDemo` si `isDemoMode()` est vrai, `DashboardSupabase` sinon.
- Les lectures Supabase sont centralisees dans `src/services/dashboard.ts`, pas directement dans la route.
- Aucune requete ne filtre manuellement par `establishment_id` : la RLS reste la source de securite et de cloisonnement.
- La route utilise `fetchDashboardData()`, qui fait un nombre fixe de lectures pour eviter les N+1.
- Les KPIs simples sont derives de `placements`, `documents`, `visits`, `alerts` et non plus de `data/demo` en prod.
- Les aggregations plus metier en P0.3 (charge profs, reseau entreprises, secteurs) sont calculees cote client depuis des datasets deja bornes par RLS.
- `AlertList` et `ActivityTimeline` demo ne sont pas reutilises en prod car leurs shapes sont couples aux types demo. J'ai ajoute des listes locales simples pour afficher `AlertRow` et `AuditLogRow`.
- Le role switcher demo de la sidebar est cache hors demo pour eviter l'effet "application non vendable" en production.

## Architecture des donnees fetched

```text
src/routes/dashboard.tsx
  -> DashboardPage
     -> isDemoMode()
        -> DashboardDemo
           -> src/data/demo.ts
        -> DashboardSupabase
           -> useAuth()
           -> fetchDashboardData()
              -> getSupabase()
              -> RLS naturelle PostgreSQL
              -> pfmp_periods / placements / visits / documents
              -> teachers / companies / alerts / audit_logs
```
